## Phase 1: 可验证闭环（第 1 周）

### 1.1 前端"一键验证"按钮

**目标**：在 Canvas 报告顶部和审计面板底部添加"验证完整性"按钮，点击后调用后端 `/verify` 端点，展示 Merkle 根比对结果。

| 任务 | 技术方案 | 文件 |
|------|----------|------|
| Verify API 客户端封装 | 新增 `verifyIntegrity(sessionId)` 函数，调用 `POST /api/v1/verify` | `frontend/src/utils/apiClient.ts` |
| Canvas 验证按钮 | 在 CanvasViewer 标题栏右侧添加"🔒 验证完整性"按钮 | `frontend/src/components/CanvasViewer.tsx` |
| 验证结果弹窗 | 新建 `VerificationModal.tsx`：展示 Merkle 根、事件数、签名状态、验证通过/失败 | `frontend/src/components/VerificationModal.tsx` |
| AuditPanel 验证入口 | 在审计面板底部"会话信息"区域添加验证按钮 | `frontend/src/components/AuditPanel.tsx` |

**验证结果展示字段**：
```typescript
interface VerifyResult {
  session_id: string;
  merkle_root: string;
  content_signature: string;
  event_count: number;
  valid: boolean;  // 比对结果
}
```

**验收标准**：
- 点击验证按钮后 2 秒内展示结果弹窗
- Merkle 根与审计面板显示的一致
- 验证失败时显示红色警告和详细差异

### 1.2 审计面板事件重放

**目标**：将"事件流"标签页从模拟构建改为真实事件流展示，支持点击单条事件查看 Merkle 路径。

| 任务 | 技术方案 | 文件 |

| 单事件详情展开 | 点击时间线节点展开 JSON payload + event_hash + 与前事件的 Merkle 路径 | `frontend/src/components/AuditPanel.tsx` |

**验收标准**：
- 事件流与 SQLite 中的 events 表完全一致
- 点击事件节点可展开查看完整 JSON 和哈希值
- 事件数量与审计元数据中的 event_count 一致

## Phase 2: 智能协同升级（第 2-3 周）

### 2.1 SACP 协议层（结构化 Agent 通信协议）

**目标**：建立正式的 SACP 协议数据模型，为 Diff 传输和上下文净化奠定基础。

| 任务 | 技术方案 | 文件 |
|------|----------|------|
| AgentMessage 模型 | 新建 `backend/app/schemas/agent_wire.py` | 新建 |
| 语义摘要字段 | Event 模型增加 `semantic_digest: str` | `backend/app/schemas/machine_wire.py` |
| TOON 序列化器 | 新建 `TOONSerializer` 类：`to_toon()` / `from_toon()` | `backend/app/services/toon_serializer.py` |
| 内容哈希升级 | `event_hash` 改为对 `payload + semantic_digest` 联合哈希 | `backend/app/infrastructure/event_store.py` |

**AgentMessage 模型设计**：
```python
class AgentMessage(BaseModel):
    msg_type: AgentMsgType  # FINDINGS_REPORT / VALIDATION_RESULT / SYNTHESIS_REQUEST
    payload: CompressedPayload
    semantic_digest: str  # 一句话语义摘要
    human_readable_ref: str | None  # 人类可读引用

class CompressedPayload(BaseModel):
    format: Literal["toon", "json", "diff"]
    data: str  # 序列化后的数据
    checksum: str  # 数据校验和
```

**验收标准**：
- TOON 序列化/反序列化正确率 100%
- 相同数据比 JSON 少 25%+ 字符
- event_hash 包含 semantic_digest

### 2.2 Diff 传输（增量更新）

**目标**：Research → Verification 不再传递完整 findings 列表，只传递变化部分。

| 任务 | 技术方案 | 文件 |
|------|----------|------|
| Finding 版本追踪 | Finding 模型增加 `revision: int` 字段 | `backend/app/schemas/machine_wire.py` |
| Diff 生成器 | 新增 `FindingDiffEngine`：比较两个 finding 列表，输出 `added/modified/unchanged/removed` | `backend/app/services/diff_engine.py` |
| Research Agent 改造 | Research Agent 输出时标记 revision；Orchestrator 在传递时计算 diff | `backend/app/agents/research.py` |
| Verification Agent 改造 | Verification Agent 接收 diff 格式，自动合并到本地状态 | `backend/app/agents/verification.py` |

**Diff 格式示例**（TOON）：
```toon
diff{
  added[1]: f4,"New finding",0.90
  modified[1]: f1,"Updated claim",0.88  # revision 1->2
  unchanged[2]: f2,f3
  removed[1]: f5
}
```

**验收标准**：
- 多轮任务中 Verification 输入 Token 比完整列表减少 60%+
- Diff 合并后 findings 状态与完整列表一致
- 100 次测试无合并错误

### 2.3 上下文净化

**目标**：每个 Agent 只接收其任务所需的最小上下文，过滤无关历史事件。

| 任务 | 技术方案 | 文件 |
|------|----------|------|
| 上下文过滤器 | 新增 `ContextFilter`：根据 Agent 类型过滤事件类型 | `backend/app/services/context_filter.py` |
| Research 上下文 | 只接收 TaskStarted + 历史对话摘要 | `backend/app/core/orchestrator.py` |
| Verification 上下文 | 只接收 TaskStarted + FindingReported（不含 LLM 原始响应） | `backend/app/core/orchestrator.py` |
| Synthesis 上下文 | 只接收 FindingValidated + ConflictDetected（不含 Research 原始数据） | `backend/app/core/orchestrator.py` |

**ContextFilter 逻辑**：
```python
AGENT_CONTEXT_MAP = {
    "research": ["TaskStarted"],
    "verification": ["TaskStarted", "FindingReported"],
    "synthesis": ["FindingValidated", "ConflictDetected", "FindingRolledBack"],
}
```

**验收标准**：
- 每个 Agent 的 Prompt 中不包含无关事件
- Synthesis 输入 Token 减少 30%+

### 2.4 模型分层

**目标**：Verification Agent 默认使用轻量模型（gpt-4o-mini），Research 和 Synthesis 保持强模型。

| 任务 | 技术方案 | 文件 |
|------|----------|------|
| 默认模型分层 | Verification 默认模型改为 gpt-4o-mini（已在 ModelSelector 支持） | `frontend/src/components/ModelSelector.tsx` |
| 后端强制分层 | Orchestrator 中如果 verification 模型未指定，自动降级为轻量模型 | `backend/app/core/orchestrator.py` |
| 成本对比记录 | 在审计面板显示"本任务因模型分层节省 $X" | `frontend/src/components/AuditPanel.tsx` |

**验收标准**：
- Verification 默认使用轻量模型
- 成本追踪正确反映不同模型的定价
- 单任务总成本降低 20%+

## Phase 3: 生产硬化（第 3-4 周）

### 3.1 HTTPS 强制

**目标**：生产环境强制 HTTPS，开发环境保持 HTTP 方便调试。

| 任务 | 技术方案 | 文件 |
|------|----------|------|
| Nginx 反向代理 | 前端 Dockerfile 中增加 Nginx 配置：80 端口重定向到 443，443 配置 SSL | `frontend/Dockerfile` + 新增 `frontend/nginx.conf` |
| 后端 HTTPS 中间件 | `main.py` 增加 `HTTPSRedirectMiddleware`：非开发环境时 HTTP 请求返回 308 | `backend/app/main.py` |
| 环境变量控制 | `ENVIRONMENT=production` 时启用强制 HTTPS | `.env.example` |
| 自签名证书脚本 | 新增 `scripts/generate-ssl.sh` 用于开发环境测试 | 新建 |

**Nginx 配置片段**：
```nginx
server {
    listen 80;
    return 308 https://$host$request_uri;
}
server {
    listen 443 ssl;
    ssl_certificate /etc/nginx/ssl/cert.pem;
    ssl_certificate_key /etc/nginx/ssl/key.pem;
}
```

**验收标准**：
- `ENVIRONMENT=production` 时 HTTP 请求自动 308 到 HTTPS
- 开发环境（`ENVIRONMENT=development`）不受影响
- Docker Compose 启动后 HTTPS 正常访问
