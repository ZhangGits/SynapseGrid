## 1. 系统概述

SynapseGrid 是一个基于自然语言与图形化交互双向反馈的 AI Agent 系统，核心采用**事件溯源（Event Sourcing）**架构，每个用户会话生成一条不可变的审计日志链（SQLite 持久化），支持分析要点的回退、版本重建和加密完整性验证。

### 1.1 核心能力

| 能力 | 说明 |
|------|------|
| 三阶段 Agent 流水线 | Research → Verification → Synthesis |
| 事件溯源审计 | 每个操作持久化为事件，支持 Merkle 完整性验证 |
| 分析要点回退 | 用户可回退任意 finding，系统基于未回退 finding 重建报告 |
| Canvas 可视化 | 结构化报告渲染（KPI、图表、溯源标签） |
| 审计面板 | 事件流、完整性验证、审计包导出 |
| 多模型支持 | OpenAI/DeepSeek/Claude/Qwen/Kimi/Gemini，路由层自动适配 |

### 1.2 技术栈

| 层级 | 技术 |
|------|------|
| 后端 | Python 3.11, FastAPI, SQLite (WAL 模式), asyncio |
| 前端 | React 18, TypeScript, Vite, TailwindCSS |
| 部署 | Docker, Docker Compose, Nginx |
| LLM | 多提供商路由（OpenAI 兼容 / Claude / Gemini） |

## 3. 后端模块详解

### 3.1 数据模型层 (`app/schemas/`)

#### 3.1.1 `Event` — 事件溯源核心

```python
class Event(BaseModel):
    version: int           # 存储分配的自增版本号
    event_type: str        # TaskStarted / FindingReported / FindingValidated / ...
    actor: LlmConfig | str # 执行者（LLM 配置或 "rollback_controller"）
    payload: dict          # 事件负载
    metadata: dict         # 元数据（cost_usd, agent_name 等）
    semantic_digest: str   # 一句话语义摘要（审计层快速理解）
    event_hash: str        # SHA-256 哈希（sha256:... 格式）
    created_at: str        # ISO 格式 UTC 时间
```

**哈希计算**（`compute_hash`）：
```
SHA256(event_type | actor | JSON(payload) | JSON(metadata) | semantic_digest | version)
```

**事件类型清单**：

| 事件类型 | 触发时机 | payload 内容 |
|----------|----------|-------------|
| `TaskStarted` | 任务开始 | prompt, template, output_mode |
| `FindingReported` | Research 生成 finding | finding 字典 |
| `FindingValidated` | Verification 通过 | finding_id |
| `ConflictDetected` | 发现矛盾 | conflict 字典 |
| `SynthesisGenerated` | Synthesis 完成 | markdown_content, canvas_schema |
| `FindingRolledBack` | 用户回退 | finding_id, reason |
| `SynthesisRestored` | 回退后重建 | 新 synthesis 结果 |

#### 3.1.2 `LlmConfig` — LLM 配置

```python
class LlmConfig(BaseModel):
    provider: str = "chatgpt"   # chatgpt/deepseek/claude/qwen/kimi/gemini/custom
    model: str = "gpt-4o-mini"
    api_key: str = ""
    base_url: str | None = None  # 自定义 API 地址
```

#### 3.1.3 `TaskRequest` / `TaskResponse`

- `TaskRequest`: API 入口，包含 user_id, prompt, 三个阶段的 LLM 配置, template, output_mode
- `TaskResponse`: 完整响应，包含 markdown_content, canvas_schema, audit_metadata, lineage_data, rollback_options

### 3.2 编排层 (`app/core/orchestrator.py`)

#### 3.2.1 `Orchestrator.run_task()` — 三阶段流水线

```python
async def run_task(self, request: TaskRequest) -> TaskResponse:
    # 1. 创建会话 → TaskStarted 事件
    # 2. ResearchAgent.run() → FindingReported 事件（每个 finding 一个事件）
    # 3. VerificationAgent.run() → FindingValidated / ConflictDetected 事件
    # 4. SynthesisAgent.run() → SynthesisGenerated 事件
    # 5. 构建 TaskResponse
```

#### 3.2.2 模型分层策略

为降低成本，Verification 阶段自动降级到轻量模型，**但保持同一 provider**（防止 api_key 不匹配导致连接失败）：

```python
LIGHT_MODEL_MAP = {
    "chatgpt": "gpt-4o-mini",
    "deepseek": "deepseek-chat",
    "claude": "claude-3-haiku",
    "qwen": "qwen-turbo",
    "kimi": "moonshot-v1-8k",
    "gemini": "gemini-1.5-flash",
}
```

#### 3.2.3 回退处理

```python
async def rollback_finding(self, ctx, finding_id: str, reason: str) -> TaskResponse:
    # RollbackController 执行回退 → FindingRolledBack 事件
    # 自动重新综合（resynthesize）→ SynthesisRestored 事件
```

### 3.3 会话管理层 (`app/core/session_manager.py`)

#### 3.3.1 `SessionContext` — 会话上下文

```python
@dataclass
class SessionContext:
    session_id: str
    event_store: SessionEventStore      # SQLite 后端
    projection: SessionStateProjection  # 内存状态
    lineage: SessionLineageGraph        # 谱系图
    write_queue: asyncio.Queue          # 单写者队列
    writer_task: asyncio.Task           # 写者协程
    created_at: datetime
```

#### 3.3.2 单写者队列模式

每个会话有一个独立的 `asyncio.Queue`，只有一个 `writer_loop` 协程接触 SQLite，避免并发写入冲突：

```python
async def _writer_loop(self, ctx: SessionContext) -> None:
    while True:
        event, future = await ctx.write_queue.get()
        if event is None: break  # 毒丸信号
        saved = ctx.event_store.append(event)
        future.set_result(saved)
```

### 3.4 事件存储层 (`app/infrastructure/event_store.py`)

#### 3.4.1 `SessionEventStore` — SQLite 后端

每个会话一个独立的 `.db` 文件，使用 WAL 模式：

```sql
PRAGMA journal_mode=WAL;
PRAGMA wal_autocheckpoint=2000;
PRAGMA cache_size=-64000;      -- 64MB 缓存
PRAGMA synchronous=NORMAL;
PRAGMA busy_timeout=5000;
```

**表结构**：
- `events`: 主事件表（version, event_type, actor, payload, metadata, semantic_digest, event_hash, created_at）
- `snapshots`: 状态快照（version, state, merkle_root, created_at）
- `lineage`: 谱系图序列化

#### 3.4.2 快照机制

每 50 个事件自动保存一次快照，加速历史状态重建：

```python
if saved.version % SNAPSHOT_INTERVAL == 0:
    ctx.event_store.save_snapshot(
        saved.version,
        ctx.projection.dump(),
        ctx.projection.merkle_root,
    )
```

### 3.5 状态投影层 (`app/core/state_machine.py`)

#### 3.5.1 `SessionStateProjection`

从事件流重建内存状态，维护：
- `task`: 任务信息
- `findings`: 分析要点字典（id → finding）
- `conflicts`: 矛盾字典
- `rolled_back_findings`: 已回退 finding ID 列表
- `costs`: 成本记录
- `synthesis`: 综合结果
- `canvas_schema`: Canvas 结构化数据

#### 3.5.2 `_project()` — 事件投影函数

根据 `event_type` 分发到不同的状态更新逻辑：

```python
def _project(event, state):
    if event.event_type == "FindingReported":
        state["findings"][finding["id"]] = finding
    elif event.event_type == "FindingRolledBack":
        state["findings"][fid]["rolled_back"] = True
    # ...
```

### 3.6 LLM 调用层 (`app/services/llm_client.py`)

#### 3.6.1 路由层设计

按 **API 格式** 分类（不是按 provider）：

| 格式类别 | 调用器 | 包含的 provider |
|----------|--------|----------------|
| OpenAI 兼容 | `_call_openai_compatible()` | chatgpt, deepseek, qwen, kimi, custom |
| Claude 原生 | `_call_claude()` | claude |
| Gemini 兼容 | `_call_gemini()` | gemini |

路由决策：
```python
if provider in OPENAI_COMPATIBLE:
    return await _call_openai_compatible(...)
elif provider == "claude":
    return await _call_claude(...)
elif provider == "gemini":
    return await _call_gemini(...)
```

#### 3.6.2 `LLMEngine` — 黑盒封装

上层唯一接口：

```python
# 构建配置（自动推断 provider）
config = LLMEngine.build_config(api_key="sk-xxx", model="deepseek-chat")

# 调用（完全无感知底层差异）
result = await LLMEngine.chat(config, system_prompt="...", user_prompt="...")
```

**provider 自动推断**（`_resolve_provider`）：
- 精确匹配：`MODEL_PROVIDER_MAP["deepseek-chat"] = "deepseek"`
- 前缀匹配：`gpt-4o-2024-08-06` → `gpt-4o` → `"chatgpt"`
- 兜底推断：`model.startswith("deepseek")` → `"deepseek"`

### 3.7 Agent 层 (`app/agents/`)

#### 3.7.1 `ResearchAgent` — 研究 Agent

- 接收 prompt + template，生成 3-5 个 finding
- 使用系统提示词要求 LLM 返回 JSON 格式
- 支持 TOON 格式（`@id|置信度|观点摘要`）作为输入

#### 3.7.2 `VerificationAgent` — 验证 Agent

- 验证 findings 的合理性
- 检测 findings 之间的逻辑矛盾
- 模型分层：自动降级到当前 provider 的轻量模型

#### 3.7.3 `SynthesisAgent` — 综合 Agent

- 根据 findings + conflicts 生成最终 Markdown 报告
- 在报告中插入溯源标签：
  - `<provenance finding="f1,f2">...</provenance>`
  - `<data value="45.2" type="percentage">...</data>`
  - `<conflict id="c1" between="f2,f3">...</conflict>`
- 支持 Canvas 模式输出

### 3.8 基础设施层 (`app/infrastructure/`)

| 模块 | 功能 |
|------|------|
| `merkle.py` | Merkle 树（SHA-256 链），用于完整性验证 |
| `rollback_controller.py` | 回退逻辑：写入 FindingRolledBack 事件，触发重新综合 |
| `lineage_graph.py` | 谱系图：记录 finding 的依赖关系 |
| `cleanup_scheduler.py` | 定时清理过期会话（默认 300 秒后删除 SQLite 文件） |

### 3.9 TOON 序列化 (`app/services/toon_serializer.py`)

**TOON (Tabular Object Notation)** 格式 — 结构化 Agent 通信格式：

```
findings[3]{id,claim,confidence}:
  f1,"NEV sales +45.2% YoY",0.92
  f2,"CATL battery cost $80/kWh",0.85
```

优势：比 JSON 节省 25-40% Token，人类可读。

### 3.10 Diff 引擎 (`app/services/diff_engine.py`)

用于 Research → Verification 的增量传输：
- 只传递变化的 finding（added/modified/removed）
- unchanged 的 finding 只传 ID
- 大幅减少 Token 消耗

## 5. 关键算法

### 5.1 Merkle 完整性验证

```
events: [e1, e2, e3, ...]
hashes: [h1, h2, h3, ...]  where hi = SHA256(ei)

Merkle Tree:
        root = Hash(h1 + h2 + h3 + ...)
       /    \
    h1+h2   h3+h4
    /   \    /   \
   h1   h2  h3   h4
```

验证时重新计算所有事件的哈希，与存储的 `event_hash` 比对，然后重建 Merkle 树比对根哈希。

### 5.2 回退算法

1. 用户请求回退 finding `f1`
2. `RollbackController` 写入 `FindingRolledBack` 事件
3. `Orchestrator.resynthesize()`：
   - 从投影获取所有未回退的 findings
   - 重新运行 `SynthesisAgent`
   - 写入 `SynthesisRestored` 事件
4. 新报告只包含未回退的 findings

### 5.3 状态重建（崩溃恢复）

```python
# 从 SQLite 读取所有事件
events = event_store.list_events()

# 完全重建
projection.rebuild(events)

# 或快照加速
snapshot = event_store.get_latest_snapshot()
state = snapshot.state
for event in events[snapshot.version:]:
    _project(event, state)
```

### 5.4 LLMEngine 路由算法

```
输入: model_name (如 "deepseek-chat")
输出: provider + API 调用器

1. 精确匹配: MODEL_PROVIDER_MAP[model_name]
2. 前缀匹配: 如 "gpt-4o-2024-08-06" 匹配 "gpt-4o"
3. 兜底推断: model.startswith("deepseek") → "deepseek"

路由:
  provider in {"chatgpt","deepseek","qwen","kimi","custom"}
    → _call_openai_compatible()
  provider == "claude"
    → _call_claude()
  provider == "gemini"
    → _call_gemini()
```

## 6. 数据流

### 6.1 正常任务流程

```
用户输入 → POST /tasks
    ↓
Orchestrator.run_task()
    ↓
创建 Session → TaskStarted 事件
    ↓
ResearchAgent.run() → LLMEngine.chat()
    ↓
FindingReported 事件（5 个）
    ↓
VerificationAgent.run() → LLMEngine.chat()
    ↓
FindingValidated 事件（5 个）+ ConflictDetected（0 个）
    ↓
SynthesisAgent.run() → LLMEngine.chat()
    ↓
SynthesisGenerated 事件
    ↓
构建 TaskResponse → 返回给前端
```

### 6.2 回退流程

```
用户点击回退 → POST /sessions/{sid}/rollback
    ↓
RollbackController.execute()
    ↓
FindingRolledBack 事件
    ↓
Orchestrator.resynthesize()
    ↓
收集未回退 findings → SynthesisAgent.run()
    ↓
SynthesisRestored 事件
    ↓
返回更新后的 TaskResponse
```

## 7. 安全设计

### 7.1 数据完整性

- 每个事件有 SHA-256 哈希
- Merkle 树根哈希可验证整条链
- 审计面板提供完整性验证按钮

### 7.2 传输安全

- 生产环境强制 HTTPS（308 重定向）
- Nginx 反向代理 + SSL 终止
- CORS 限制

### 7.3 访问控制

- 速率限制（基于 IP）
- 请求大小限制（10MB）
- 环境变量控制 HMAC 密钥

### 7.4 审计安全

- 审计包导出（ZIP，加密存储）
- SQLite WAL 模式保证崩溃一致性
