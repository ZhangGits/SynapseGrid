## 1. 系统概述

### 1.1 项目定位

SynapseGrid 是一个**可审计的事件溯源 AI Agent 系统**，采用三阶段 Agent 流水线（Research → Verification → Synthesis）执行分析任务。核心创新在于：

- **事件溯源（Event Sourcing）**: 所有 Agent 操作均记录为不可变事件，支持完整审计追溯
- **Merkle 树完整性验证**: 事件链通过 Merkle 树保证数据完整性
- **HMAC 内容签名**: 报告内容使用 HMAC-SHA256 签名防止篡改
- **Ed25519 签名（可选）**: 支持第三方审计签名
- **可回退机制**: 支持对单个分析要点（Finding）进行回退，并基于剩余 findings 自动重建报告
- **双输出模式**: 支持 Markdown 和 Canvas（结构化可视化 — LogicGraph 论证结构图）两种输出格式
- **多模型支持**: 统一 LLM 调用层，支持 OpenAI、Claude、DeepSeek、Qwen、Moonshot、Gemini 及自定义提供商

### 1.2 技术栈

| 层级 | 技术 | 用途 |
|------|------|------|
| 后端框架 | FastAPI (Python 3.11+) | REST API 服务 |
| 数据存储 | SQLite (WAL 模式) | 事件存储、快照、谱系图 |
| 异步运行时 | asyncio | 单写者队列、并发处理 |
| LLM 客户端 | httpx (异步 HTTP) | 统一调用 OpenAI/Claude/DeepSeek 等 |
| Token 计数 | tiktoken (可选) | 精确 Token 计数 |
| 加密 | PyNaCl (可选) | Ed25519 签名 |
| 前端框架 | React 18 + TypeScript | SPA 用户界面 |
| 构建工具 | Vite | 前端构建与开发服务器 |
| Markdown 渲染 | react-markdown | 报告展示 |
| 容器化 | Docker + Docker Compose | 部署运行 |

### 1.3 核心概念

| 概念 | 说明 |
|------|------|
| **Event（事件）** | 不可变的日志条目，记录 Agent 的每次操作，通过 SHA-256 哈希保护完整性 |
| **Session（会话）** | 一次分析任务的完整生命周期，每个会话一个独立的 SQLite 数据库 |
| **Finding（分析要点）** | Research Agent 生成的单个分析结论，包含 id、claim、evidence、confidence |
| **CredibilityAssessment（可信度评估）** | Verification Agent 对 finding 的四维度独立评估 |
| **FindingRelation（关系）** | 两个 finding 之间的关系（视角差异/张力/真实矛盾） |
| **LogicGraph（论证结构图）** | Synthesis Agent 提取的节点-边结构图，用于 Canvas 渲染 |
| **LogicNode / LogicEdge** | 论证结构图的基本单元（question/conclusion/claim/evidence） |
| **Merkle Root（Merkle 根）** | 事件链的完整性校验和 |
| **Semantic Digest（语义摘要）** | 每个事件的一句话描述，用于审计层快速理解 |
| **SACP 协议** | 结构化 Agent 通信协议，支持 TOON/JSON/Diff 三种压缩载荷格式 |

### 2.2 分层架构说明

系统采用严格的分层架构，自上而下分为 7 层：

| 层 | 目录 | 职责 | 依赖方向 |
|----|------|------|----------|
| **中间件层** | `main.py` | CORS、HTTPS、访问日志、请求大小限制、速率限制 | → 无 |
| **API 路由层** | `app/api/v1/` | REST 端点定义、请求验证、响应构建 | → 编排层 |
| **编排层** | `app/core/orchestrator.py` | 三阶段流水线协调、回退处理、成本估算 | → Agent 层、会话管理层 |
| **会话管理层** | `app/core/session_manager.py` | 会话生命周期、单写者队列、快照调度、清理 | → 基础设施层 |
| **Agent 层** | `app/agents/` | Research、Verification、Synthesis 三个 Agent | → LLM 引擎层 |
| **LLM 引擎层** | `app/services/llm_client.py` | 多提供商路由、统一调用接口、Token 追踪 | → 外部 API |
| **基础设施层** | `app/infrastructure/` | 事件存储、Merkle 树、回退控制、谱系图、清理 | → 文件系统 / SQLite |

### 2.3 事件溯源架构

```
                         ┌──────────────────────┐
                         │    Client (前端)      │
                         └──────────┬───────────┘
                                    │ HTTP POST /tasks
                                    ▼
                         ┌──────────────────────┐
                         │    Orchestrator      │
                         │  (三阶段流水线)       │
                         └──────────┬───────────┘
                                    │ append_event()
                                    ▼
                         ┌──────────────────────┐
                         │   SessionManager     │
                         │  ┌────────────────┐  │
                         │  │ write_queue    │  │
                         │  │ .put(event)    │  │
                         │  └───────┬────────┘  │
                         │          ▼           │
                         │  ┌────────────────┐  │
                         │  │ _writer_loop() │  │ ← 唯一写者协程
                         │  └───────┬────────┘  │
                         │          │           │
                         └──────────┼───────────┘
                                    │
                    ┌───────────────┼───────────────┐
                    ▼               ▼               ▼
            ┌──────────────┐ ┌──────────────┐ ┌──────────────┐
            │ EventStore   │ │ Projection   │ │ LineageGraph │
            │ (SQLite WAL) │ │ (内存状态)   │ │ (谱系图)     │
            │              │ │              │ │              │
            │ INSERT INTO  │ │ _project()   │ │ apply()      │
            │  events ...  │ │ 更新 state   │ │ 更新图结构   │
            └──────────────┘ └──────────────┘ └──────────────┘
                    │
                    │ 每 50 个事件
                    ▼
            ┌──────────────┐
            │   Snapshot   │
            │ (版本X状态)  │
            └──────────────┘
```

**关键设计原则**:
1. **先持久化再投影**: 事件先写入 SQLite 再更新内存状态，确保数据不丢失
2. **单写者队列**: 每个会话一个 `asyncio.Queue` + 一个 `_writer_loop` 协程，避免 SQLite 并发写入冲突
3. **不可变事件**: 事件一旦写入即不可修改，所有状态通过重放事件重建
4. **快照加速**: 每 50 个事件保存状态快照，历史查询时从最近快照开始重放

### 3.1 数据模型层 (`app/schemas/`)

#### 3.1.1 `Event` — 事件溯源核心不可变单元

文件: `backend/app/schemas/machine_wire.py`

```python
class Event(BaseModel):
    version: int = 0                    # 自增版本号（由 EventStore 分配）
    event_type: str = ""                # 事件类型
    actor: LlmConfig | str = LlmConfig() # 执行者（LLM 配置或字符串如 "rollback_controller"）
    payload: dict[str, Any] = {}        # 事件负载（JSON 可序列化数据）
    metadata: dict[str, Any] = {}       # 元数据（user_id, agent_name, cost_usd 等）
    semantic_digest: str = ""           # 一句话语义摘要（审计层快速理解）
    event_hash: str = ""                # SHA-256 哈希（前缀 "sha256:"）
    created_at: str = ""                # ISO 格式 UTC 时间
```

**哈希计算算法** (`compute_hash`):
```python
raw = f"{event_type}|{actor_json}|{payload_json}|{metadata_json}|{semantic_digest}|{version}"
event_hash = f"sha256:{sha256(raw.encode('utf-8')).hexdigest()}"
```

#### 3.1.5 LogicGraph (论证结构图)

```python
class LogicNode(BaseModel):
    id: str                             # 节点唯一标识
    type: str                           # question | conclusion | claim | evidence
    label: str                          # 完整文本
    summary: str | None = None          # 1-2句摘要（前端优先显示）
    finding_id: str | None = None       # 关联的 finding ID
    credibility: float = 0.5            # 可信度 (0-1)
    rolled_back: bool = False           # 是否已回退
    importance: float = 0.5             # 重要性权重 (0-1)

class LogicEdge(BaseModel):
    from_id: str                        # 源节点 ID
    to_id: str                          # 目标节点 ID
    type: str                           # supports | perspective_difference | tension | genuine_contradiction
    description: str | None = None      # 关系描述

class LogicGraph(BaseModel):
    nodes: list[LogicNode]
    edges: list[LogicEdge]
```

#### 3.1.6 SACP 协议模型 (`agent_wire.py`)

```python
class AgentMsgType(str, Enum):
    FINDINGS_REPORT = "findings_report"      # Research → Verification
    VALIDATION_RESULT = "validation_result"  # Verification → Synthesis
    SYNTHESIS_REQUEST = "synthesis_request"  # Orchestrator → Synthesis
    TASK_STARTED = "task_started"
    ROLLBACK_NOTICE = "rollback_notice"

class CompressedPayload(BaseModel):
```

#### 3.3.3 单写者队列模式

```
append_event(ctx, event)
    │
    ├─ 1. future = create_future()
    ├─ 2. write_queue.put((event, future))
    │
    ▼
_writer_loop(ctx)  ← 每个会话唯一写者协程
    │
    ├─ event, future = await write_queue.get()
    ├─ saved = event_store.append(event)   ← 分配 version + hash
    ├─ future.set_result(saved)            ← 通知调用方
    │
    ▼
append_event 继续:
    ├─ 3. projection.apply(saved)          ← 更新内存投影
    ├─ 4. lineage.apply(saved)             ← 更新谱系图
    ├─ 5. lineage.persist_to_store()       ← 谱系图持久化
    └─ 6. 每 50 个事件保存快照
```

**毒丸信号**: `write_queue.put((None, None))` 触发 writer_loop 退出

**异常恢复**: writer_loop 崩溃时自动重启（最多 3 次），超过则通知所有等待 future

### 3.5 Agent 层 (`app/agents/`)

#### 3.5.1 ResearchAgent

**职责**: 将用户问题拆解为带 `<finding>` 标签的 Markdown 叙述

**`run(prompt, llm, template, conversation_history) → (raw_markdown, findings)`**:

1. 构建系统提示词 `FINDINGS_MARKDOWN_PROMPT`（要求输出 `<finding id="f1" confidence="0.85">...</finding>`）
2. 通过 `template_engine.render_template()` 加载场景模板上下文
3. 调用 `LLMEngine.build_config()` → `call_llm()` 获取 LLM 响应
4. `parse_finding_tags()` 用正则解析 `<finding>` 标签 → findings 列表
5. 如果没有匹配到标签，尝试 JSON fallback 解析
6. 如果 LLM 调用失败或 API key 为空，降级到 mock 模式

**`rebuild(original_markdown, rolled_back_id, reason, remaining_findings, llm) → str`**:
- 在回退时调用，用 LLM 或纯文本方式重建 Markdown

**Mock 模式**: 根据 template 类型 (general/finance/legal) 生成确定性 mock findings

#### 3.5.2 VerificationAgent

**职责**: 对 findings 进行四维度独立可信度评估 + 检测关系

**`run(findings, llm, template) → (assessments, relations)`**:

1. 将 findings 转换为 TOON 格式: `@f1|85%|claim_text...`
2. `CREDIBILITY_SYSTEM_PROMPT` 要求 LLM 返回 JSON:
   ```json
   {
     "assessments": [{
       "finding_id": "f1",
       "evidence_strength": 0.85, "source_reliability": 0.80,
       "reasoning_soundness": 0.90, "data_consistency": 0.88,
       "overall_credibility": 0.86,
       "assessment": "论据充分，推理严密",
       "concerns": ["数据来源可能过时"],
       "suggestions": ["建议补充最新数据"]
     }],
     "relations": [{
       "between": ["f1", "f2"],
       "relation_type": "perspective_difference",
       "description": "f1从长期技术角度分析..."
     }]
   }
   ```
3. 解析 JSON → CredibilityAssessment 对象列表 + FindingRelation 对象列表

**设计理念 (v4.0)**: 
- 不再"检测矛盾并标记无效" → 矛盾可能是视角差异
- 三个关系类型: `perspective_difference`（视角差异）、`tension`（张力可调和）、`genuine_contradiction`（真实矛盾）

#### 3.5.3 SynthesisAgent

**职责**: 从报告和评估中提取 LogicGraph 论证结构 + 为节点生成摘要

**`run(raw_markdown, findings, assessments, relations, llm, template) → SynthesisResult`**:

1. 构建 credibility map + findings 摘要 + relations 摘要
2. `LOGIC_GRAPH_SYSTEM_PROMPT` 要求 LLM 返回 JSON (nodes + edges)
3. 解析 → LogicNode 列表 + LogicEdge 列表
4. **节点摘要生成**: 
   - `_generate_node_summaries()`: 调用 LLM 为每个节点生成 1-2 句摘要
   - `_generate_local_summaries()`: 本地 fallback（启发式规则从 label 截取）
5. 构建 `node_map`（节点 ID → finding ID 映射）
6. 返回 `SynthesisResult(logic_graph, enriched_markdown, node_map)`

4. INSERT INTO events
5. COMMIT
6. 返回已分配 version/hash 的事件

#### 3.7.2 `MerkleTree` — 完整性验证

**构建算法**:
```python
class MerkleTree:
    def append(self, event_hash):
        self._leaves.append(event_hash)
        self._recompute_root()

    def _recompute_root(self):
        nodes = self._leaves[:]
        while len(nodes) > 1:
            next_level = []
            for i in range(0, len(nodes), 2):
                if i + 1 < len(nodes):
                    combined = nodes[i] + nodes[i + 1]
                else:
                    combined = nodes[i] + nodes[i]  # 奇数节点复制自身
                h = sha256(combined.encode()).hexdigest()
                next_level.append(f"sha256:{h}")
            nodes = next_level
        self._root_hash = nodes[0]
```

**Ed25519 签名 (可选)**:
- 依赖 PyNaCl 库 (`nacl.bindings`)
- `sign_content(content)`: 使用 Ed25519 私钥签名
- `verify_content(content, signature, public_key)`: 验证签名
- `get_public_key_hex()`: 获取公钥

## 5. 关键算法与数据结构

### 5.1 事件溯源模式 (Event Sourcing)

```
写入路径:
  append_event() → write_queue → _writer_loop() → SQLite INSERT
    → future.set_result(saved) → projection.apply() → lineage.apply()
    → 每 50 个事件 → save_snapshot()

查询路径:
  直接: projection.dump() → 当前完整状态
  历史: get_state_at(version) → 最近快照 + 增量事件重放
  重建: projection.rebuild(events) → 完全重放

验证路径:
  Merkle 根 = MerkleTree(所有 event_hash).root_hash
  HMAC 签名 = hmac(HMAC_KEY, session_id|merkle_root|event_count|markdown)
```

### 5.2 Merkle 树算法

```
构建过程:
  leaves = [hash(event1), hash(event2), hash(event3), ...]
  
  while len(leaves) > 1:
    next_level = []
    for i in range(0, len(leaves), 2):
      if i+1 < len(leaves):
        h = sha256(leaves[i] + leaves[i+1])
      else:
        h = sha256(leaves[i] + leaves[i])  # 奇数节点复制自身
      next_level.append(f"sha256:{h}")
    leaves = next_level
  
  merkle_root = leaves[0]

验证:
  重新计算 Merkle 根 → 与存储的值比较 → 一致则完整性通过
```

### 5.3 HMAC-SHA256 内容签名

```
输入: session_id, merkle_root, event_count, markdown_content
密钥: SYNAPSEGRID_HMAC_KEY (环境变量, ≥32 字节)

msg = f"{session_id}|{merkle_root}|{event_count}|{markdown}"
signature = "hmac:" + hmac.new(key, msg.encode(), sha256).hexdigest()
```

### 5.4 SHA-256 事件哈希

```
输入: event_type, actor_json, payload_json, metadata_json, semantic_digest, version

raw = f"{event_type}|{actor_json}|{payload_json}|{metadata_json}|{semantic_digest}|{version}"
event_hash = f"sha256:{sha256(raw.encode('utf-8')).hexdigest()}"
```

### 5.5 TOON 序列化格式

```
findings[N]{field1,field2,...}:
  value1,value2,...
  value3,value4,...

比 JSON 节省约 25-40% Token。通过去除 JSON 引号、缩进和重复字段名实现。
```

### 5.6 Diff 增量更新

```
diff{
  added[N]: <TOON 行>
  modified[N]: <TOON 行>
  unchanged[N]: id1,id2,...
  removed[N]: id1,id2,...
}

apply_diff(base_findings, diff) → 更新后的 findings 列表
```

### 5.7 上下文净化 (ContextFilter)

每个 Agent 只接收最小必要上下文:

| Agent | 接收事件类型 | Token 节省 |
|-------|-------------|-----------|
| Research | TaskStarted | 约 60% |
| Verification | TaskStarted + FindingReported | 约 40% |
| Synthesis | FindingValidated + ConflictDetected + FindingRolledBack | 约 30% |

### 5.8 可信度加权平均

```
overall_credibility = 
    evidence_strength × 0.3 
  + source_reliability × 0.2 
  + reasoning_soundness × 0.3 
  + data_consistency × 0.2
```

### 5.9 Token 计数

```python
def count_tokens(text, model):
    if tiktoken available:
        encoding = tiktoken.encoding_for_model(model)
        return len(encoding.encode(text))
    else:
        return max(1, len(text) // 3)  # 中英文混合约 3 字符/Token
```

### 5.10 成本计算

```python
def estimate_cost(model, tokens_input, tokens_output):
    pricing = MODEL_PRICING.get(model, {"input": 0.001, "output": 0.002})
    return (tokens_input * pricing["input"] + tokens_output * pricing["output"]) / 1000
```

### 5.11 单写者队列模式

```
writer_loop(ctx):
  while True:
    event, future = await write_queue.get()
    if event is None: break  # 毒丸
    try:
      saved = event_store.append(event)
      future.set_result(saved)
    except:
      future.set_exception(exc)
    
  异常恢复: 最多重启 3 次，超过则通知所有等待 future
```

### 5.12 快照-增量重放

```
get_state_at(target):
  snapshot = store.get_latest_snapshot(max_version=target)
  state = snapshot.state (深拷贝) 或 空状态
  start = snapshot.version if snapshot else 0
  tail = store.get_events(from=start+1, to=target)
  for event in tail: _project(event, state)
  return state
```

## 6. 数据流分析

### 6.1 完整任务数据流

```
用户输入 prompt
  │
  ▼
前端 POST /api/v1/tasks (TaskRequest)
  │
  ▼
Orchestrator.run_task(request)
  │
  ├─ 1. SessionManager.create_session()
  │     └─ SQLite 数据库 + write_queue + writer_loop + projection + lineage
  │
  ├─ 2. Event(TaskStarted) → append_event
  │     └─ event_store.append() → projection.apply() → lineage.apply()
  │
  ├─ 3. ResearchAgent.run(prompt, llm, template)
  │     ├─ LLMEngine.build_config() → call_llm() → LLM API
  │     ├─ parse_finding_tags() → findings
  │     └─ Event(FindingReported) × N → append_event × N
  │
  ├─ 4. 模型降级: verification model → light model
  │
  ├─ 5. VerificationAgent.run(findings, llm_light, template)
  │     ├─ TOON 格式化 findings
  │     ├─ call_llm() → {assessments, relations}
  │     ├─ Event(CredibilityAssessed) × N → append_event × N
  │     └─ Event(RelationDetected) × N → append_event × N
  │
  ├─ 6. 根据 output_mode 分支:
  │     │
  │     ├─ markdown 模式:
  │     │   PostProcessor.enrich() → 注入 provenance + 可信度摘要
  │     │   Event(MarkdownGenerated) → append_event
  │     │
  │     └─ canvas 模式:
  │         SynthesisAgent.run() → LogicGraph + node_map
  │         Event(SynthesisGenerated) → append_event
  │
  └─ 7. _build_response(ctx, request) → TaskResponse
        ├─ 提取 Synthesis/Markdown 事件
        ├─ with_audit_header() → HMAC 签名
        ├─ 构建 AuditMetadata (findings, costs, merkle_root)
        ├─ 构建 LineageData
        └─ 构建 RollbackOptions
            │
            ▼
前端接收 TaskResponse
  ├─ 更新 messages[]
  ├─ 侧边栏显示审计面板 / 画布模式
  └─ 用户可查看报告、回退 finding、验证完整性
```

### 6.2 回退数据流

```
用户点击"回退"按钮
  │
  ▼
前端 POST /tasks/{id}/rollback
  │
  ▼
Orchestrator.rollback_finding(ctx, finding_id, reason)
  │
  ├─ RollbackController.execute(ctx, finding_id, reason)
  │   ├─ 验证 finding 存在且未回退
  │   ├─ Event(FindingRolledBack) → append_event
  │   ├─ projection: findings[id]["rolled_back"] = True
  │   ├─ ResearchAgent.rebuild() → 重建 Markdown
  │   └─ 根据 output_mode 分支:
  │       ├─ markdown: MarkdownRestored 事件
  │       └─ canvas: SynthesisAgent.run() → SynthesisRestored 事件
  │
  └─ _build_response() → TaskResponse
      (已删除被回退 findings 的报告)
        │
        ▼
前端更新消息 + 审计面板 + 画布
```

### 6.3 重新综合数据流

```
用户确认重新综合
  │
  ▼
前端 POST /tasks/{id}/resynthesize
  │
  ▼
Orchestrator.resynthesize(ctx)
  │
  ├─ 从投影获取未回退 findings
  ├─ 从事件中获取 synthesis LLM 配置
  ├─ SynthesisAgent.run(active_findings, ...) → LogicGraph
  ├─ Event(SynthesisGenerated) → append_event
  └─ _build_response() → TaskResponse
```

### 6.4 完整性验证数据流

```
前端 POST /verify/simple
  │
  ▼
verify_simple(request)
  ├─ 获取会话 events
  ├─ 获取 merkle_root
  ├─ 获取最新 synthesis → markdown_content
  ├─ _sign_content(markdown, session_id, merkle_root, event_count)
  │   → HMAC-SHA256 签名
  └─ 返回 SimpleVerifyResponse { valid, session_id, merkle_root, signature }
```
