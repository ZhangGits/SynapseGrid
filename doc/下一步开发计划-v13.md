## 二、重新排定的开发计划

### 核心原则

1. **安全优先**：P0 安全修复必须先做，否则不能上线
2. **用户可见优先**：交互式 Canvas 和审计面板改造用户最能感知
3. **降本优先**：Agent 结构化通信直接降低 Token 消耗
4. **避免过度设计**：砍掉当前 MVP 阶段不必要的功能

### Phase 0: 紧急安全修复（第 1 周）

> 目标：修复阻碍生产化的底线安全问题

| # | 任务 | 文件 | 工作量 | 说明 |
|---|------|------|--------|------|
| 0.1 | **CORS 限制** | `main.py:47` | 0.5天 | `allow_origins=[os.getenv("FRONTEND_URL", "http://localhost:5173")]` |
| 0.2 | **HMAC 密钥强制校验** | `post_processor.py:25` | 0.5天 | 启动时检查密钥长度 ≥ 32 字节，默认值则拒绝启动 |
| 0.3 | **前端 API 地址环境变量化** | `apiClient.ts:15` | 0.5天 | 改为 `import.meta.env.VITE_API_BASE` |
| 0.4 | **死代码清理** | 多处 | 1天 | 移除 `budget_usd`、`branchSession`、`process_feedback` 相关代码 |
| 0.5 | **统一定价表** | `orchestrator.py:507-568` | 0.5天 | 三处定价表合并引用 `cost_tracker.MODEL_PRICING` |

**验收标准**：`docker-compose up` 后，非配置域名访问返回 CORS 错误；未设置 HMAC 密钥时后端拒绝启动；`grep -r "budget_usd\|branchSession" backend/ frontend/` 无结果。

### Phase 1: 交互式报告（第 2-4 周）

> 目标：将 Canvas 从"静态仪表盘"升级为"交互式叙事文档"，审计面板支持 Finding 卡片和事件流时间线

#### 1.1 Synthesis Agent 标记化（5 天）

**当前问题**：Synthesis Agent 输出纯 Markdown，无法将结论与原始 Finding 关联。

**改造方案**：
1. 修改 `synthesis.py` 的系统提示词，要求 LLM 输出带 `<provenance>` 标记的 Markdown
2. 增加后置校验：正则提取 provenance 标签，验证引用的 finding_id 存在
3. 标记化输出存入 `enriched_markdown` 字段，纯文本版本保留在 `markdown_content`

**标记规范**：
```markdown
<provenance finding="f1,f2">根据最新数据，新能源汽车销量同比增长45.2%</provenance>
<data value="45.2%" type="percentage">45.2%</data>
<conflict id="c1" between="f2,f3">关于电池成本的预测存在分歧</conflict>
```

#### 1.2 Canvas 渲染引擎重构（7 天）

**当前问题**：CanvasViewer 是静态组件列表，不支持交互。

**改造方案**：
1. 在 `CanvasViewer.tsx` 中增加自定义组件映射：
   - `<provenance>` → 可点击下划线 + 🔍 角标
   - `<conflict>` → 警告色边框卡片
   - `<data>` → 高亮数字
2. 内联证据展开：点击 provenance 标签时，内联展开轻量证据卡片
3. 章节导航：从 enriched_markdown 的标题结构自动生成左侧 Outline

**关键约束**：
- Canvas 中绝不出现 Finding 卡片网格
- Finding ID 不显示在文本中，只存在于 DOM 的 `data-finding-id` 属性
- 所有技术细节收敛到"点击溯源"一个入口

#### 1.3 审计面板改造（5 天）

**当前问题**：审计面板只显示扁平列表，缺乏交互性。

**改造方案**：
1. 新建 `AuditFindingCard.tsx`：每张卡片包含主张、证据、元数据、操作按钮
2. 新建 `AuditTimeline.tsx`：垂直时间线展示 Event 序列
3. 新建 `AuditAgentTrace.tsx`：按 Agent 分组展示执行追踪
4. 审计面板改为三个标签页：Findings / 事件流 / Agent 轨迹

#### 1.4 双向联动（3 天）

**目标**：实现 Canvas ↔ Audit 的穿透导航。

1. Canvas → Audit：点击 provenance 标签时，右侧面板自动切换并高亮对应 Finding
2. Audit → Canvas：点击"在报告中定位"时，Canvas 滚动到该段落并闪烁高亮

---

### Phase 2: 智能协同（第 5-7 周）

> 目标：Agent 间通信结构化，回退算法基于状态重建

#### 2.1 Agent 结构化通信（7 天）

**当前问题**：Agent 间使用自然语言传递数据，Token 消耗大。

**改造方案**：
1. Research → Verification：Research 输出序列化为 TOON 格式
2. Verification → Synthesis：Verification 输出结构化 JSON
3. Synthesis 输入改造：Prompt 中 findings 以结构化形式注入

**预期效果**：Verification 和 Synthesis 的输入 Token 减少 50%+

#### 2.2 回退算法重构（5 天）

**当前问题**：回退基于文本过滤（`rollback_controller.py` 中的 `_rebuild_synthesis`），极易失效。

**改造方案**：
1. `rollback_finding` 不再做文本过滤
2. 改为：写入 `FindingRolledBack` 事件 → 从 Projection 获取未回退的 findings → 调用 `SynthesisAgent.run()` 重新生成
3. 在报告中自动插入 Rollback Notice

#### 2.3 版本时间轴（4 天）

**目标**：支持浏览任意历史报告版本。

1. 每个 `SynthesisGenerated` 和 `SynthesisRestored` 事件视为一个报告版本
2. 在 `state_machine.py` 中增加 `get_state_at_version(version)` 
3. 前端 Audit Panel 增加"版本时间轴"侧边栏

---

### Phase 3: 生产硬化（第 8-10 周）

> 目标：部署可运维、有基本防护的生产系统

#### 3.1 SQLite 生产优化（3 天）

1. WAL 参数调优：`PRAGMA wal_autocheckpoint=2000; PRAGMA cache_size=-64000`
2. 写者协程异常捕获和自动重启
3. 磁盘水位检查

#### 3.2 审计包导出（3 天）

新增 `POST /tasks/{session_id}/export-audit`，返回 ZIP 包含：
- `events.jsonl`（完整事件流）
- `merkle_proof.json`（Merkle 路径）
- `signature.sig`（HMAC 签名）
- `report.md`（人类可读报告）

#### 3.3 Docker 部署优化（3 天）

1. 持久化存储：`/tmp/synapse_sessions` 改为命名卷
2. 健康检查端点增强
3. 日志轮转配置

#### 3.4 速率限制与防护（2 天）

1. API 限流：`slowapi` 限制 `POST /api/v1/tasks` 为 10/分钟/IP
2. 请求体限制：prompt 长度限制 8000 字符
3. 成本告警：单 Session 成本上限

#### 3.5 集成测试（4 天）

1. 端到端测试：创建任务 → 验证 Merkle 链 → 回退 → 重新综合 → 导出审计包
2. 前端测试：Playwright 测试 Canvas 交互
3. 性能基准记录
