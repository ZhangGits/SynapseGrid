## 1. 系统概述

### 1.1 项目定位

SynapseGrid 是一个**可审计的多 Agent 协同分析系统**，采用**事件溯源 (Event Sourcing)** 架构，通过三阶段 Agent 流水线对用户提出的分析问题进行深度分析，生成带完整溯源链的可验证报告。

### 1.2 核心特性

| 特性 | 描述 |
|------|------|
| **事件溯源** | 所有状态变更以不可变事件形式持久化，支持完整回放 |
| **Merkle 完整性验证** | 事件链附带 Merkle 树根哈希 + HMAC 签名 |
| **三阶段 Agent 流水线** | Research → Verification → Synthesis |
| **双输出模式** | Markdown 模式（轻量）、Canvas 模式（论证结构图） |
| **Finding 回退** | 支持回退分析要点并重新综合报告 |
| **多 LLM 提供商** | ChatGPT / Claude / DeepSeek / Qwen / Kimi / Gemini |
| **模板系统** | YAML 场景模板（通用/金融/法律） |

### 1.3 技术栈

| 层 | 技术 |
|----|------|
| 后端框架 | Python 3.12 + FastAPI |
| 前端框架 | React 18 + TypeScript + Vite |
| 数据库 | SQLite (WAL 模式，每会话独立) |
| LLM 通信 | httpx 异步 HTTP 客户端 |
| 容器化 | Docker Compose |
| 前端 Nginx | Nginx (反向代理 + SSL 终止) |

```
└──────────────────────────────────────┘
```

### 2.3 事件溯源架构

系统核心采用事件溯源模式：

1. **写路径**：所有操作产生 Event → 写入 SQLite EventStore → 投影到内存 StateProjection → 更新 MerkleTree
2. **读路径**：从 StateProjection 直接读取当前内存状态
3. **历史查询**：通过快照 + 尾部事件重建历史状态
4. **完整性验证**：Merkle 根哈希 + HMAC 签名确保内容不被篡改

## 3. 模块设计

### 3.1 核心层 (core)

#### 3.1.1 Orchestrator（任务编排器）

**文件**: `backend/app/core/orchestrator.py`

**职责**: 协调三阶段 Agent 流水线的中央控制器。

**核心流程**:

```
TaskRequest → create_session() → TaskStarted Event
  → ResearchAgent.run() → (raw_markdown, findings)
  → 写入 FindingReported Events
  → VerificationAgent.run() → (assessments, relations)
  → 写入 CredibilityAssessed + RelationDetected Events
  → output_mode 分支:
      ├─ markdown → PostProcessor.enrich() → MarkdownGenerated Event
      └─ canvas → SynthesisAgent.run() → SynthesisGenerated Event
  → _build_response() → TaskResponse
```

**关键方法**:

| 方法 | 说明 |
|------|------|
| `run_task(request)` | 执行完整的三阶段流水线 |
| `rollback_finding(ctx, finding_id, reason)` | 回退指定 finding 并重新生成报告 |
| `resynthesize(ctx)` | 基于当前活跃 findings 重新执行 Synthesis |
| `_build_response(ctx, request)` | 从投影状态构建 API 响应 |

**双路径设计 (v4.0)**:
- **Markdown 模式**: Research → Verification → PostProcessor.enrich()（跳过 Synthesis），生成 MarkdownGenerated 事件
- **Canvas 模式**: Research → Verification → SynthesisAgent.run()，生成 SynthesisGenerated 事件（含 LogicGraph）

**模型分层策略**: Verification Agent 自动使用轻量模型（如 gpt-4o-mini 替代 gpt-4o），以降低成本。

#### 3.1.2 SessionManager（会话管理器）

**文件**: `backend/app/core/session_manager.py`

**职责**: 管理会话的完整生命周期——创建、查询、追加事件、关闭、清理。

**关键设计**:
- **会话级隔离**: 每个会话一个独立的 SQLite 数据库文件
- **单写者队列**: 每个会话一个 `asyncio.Queue`，确保事件写入串行化
- **谱系图持久化**: 每次事件追加后，谱系图同步序列化到 SQLite 的 `lineage` 表

**SessionContext 数据结构**:
```python
@dataclass
class SessionContext:
    session_id: str              # 唯一标识
    event_store: SessionEventStore  # SQLite 事件存储
    projection: SessionStateProjection  # 内存状态投影
    lineage: SessionLineageGraph  # 谱系图
    write_queue: asyncio.Queue   # 单写者队列
    writer_task: asyncio.Task    # 写者协程
    created_at: datetime         # 创建时间
```

**writer_loop 关键设计**:
- 从队列取出 (event, future) 元组
- 调用 `event_store.append(event)` 持久化事件
- 设置 `future.set_result(saved)` 通知调用方
- 收到毒丸信号 (None, None) 时退出
- 崩溃自动重启（最多 3 次）

**快照策略**: 每 50 个版本 (`SNAPSHOT_INTERVAL = 50`) 保存一次状态快照，加速历史状态查询。

**会话清理**: 通过 `CleanupScheduler` 实现延迟清理（默认 300 秒），支持持久化调度和启动补偿。

#### 3.1.3 StateMachine（状态投影）

**文件**: `backend/app/core/state_machine.py`

**职责**: 将事件流投影为可查询的内存状态。

**状态字典结构**:
```python
self.state = {
    "task": None,                    # 任务信息
    "findings": {},                  # {finding_id: finding_dict}
    "conflicts": {},                 # {conflict_id: conflict_dict}
    "credibility_assessments": {},   # {finding_id: assessment_dict}
    "relations": [],                 # [relation_dict, ...]
    "rolled_back_findings": [],      # [finding_id, ...]
    "costs": [],                     # [{agent, model, usd, tokens}, ...]
    "synthesis": None,               # Synthesis 结果
    "agent_traces": [],              # Agent 执行追踪
    "canvas_schema": None,           # 画布结构
}
```

**投影函数 `_project(event, state)`** 根据事件类型更新状态：

| 事件类型 | 状态更新 |
|---------|---------|
| TaskStarted | 存储 task 信息 |
| FindingReported | 添加到 findings 字典 |
| FindingRolledBack | 标记 finding 已回退 |
| CredibilityAssessed | 存储可信度评估 |
| RelationDetected | 添加 finding 间关系 |
| SynthesisGenerated | 存储综合结果和 canvas_schema |
| MarkdownGenerated | 存储 Markdown 模式结果 |
| SynthesisRestored / MarkdownRestored | 存储恢复结果 |

**快照加速**: `get_state_at(target_version, event_store)` 从最近的快照 + 尾部事件重建，避免全量重放。

### 3.2 Agent 层 (agents)

#### 3.2.1 ResearchAgent（研究 Agent）

**文件**: `backend/app/agents/research.py`

**职责**: 将用户 prompt 拆解为多个分析要点 (findings)，生成带 `<finding>` 标签的 Markdown 叙述。

**核心流程**:
1. 构建消息：system prompt (`FINDINGS_MARKDOWN_PROMPT`) + 渲染后的模板 + 用户 prompt
2. 调用 `LLMEngine.chat()` → 获得 LLM 响应
3. `parse_finding_tags()` 解析 `<finding id="f1" confidence="0.85">...</finding>` 标签
4. 计算成本并摊分到每个 finding
5. 返回 `(raw_markdown, findings)`

**`parse_finding_tags()` 解析策略**:
1. 主路径：正则匹配 `<finding id="..." confidence="...">...</finding>`
2. JSON 降级：尝试直接解析 JSON，或从 ` ```json ... ``` ` 代码块提取
3. 解读失败：返回空 findings 列表

**`rebuild()` 方法** — 回退后重新组织 Markdown：
1. **LLM 路径**: 使用 `REBUILD_PROMPT` 提示 LLM 基于剩余 findings 重新生成报告
2. **文本降级**: 正则移除被回退的 `<finding>` 标签，在顶部添加回退通知

**Mock 模式**: 当未配置 API Key 或 LLM 调用失败时，根据模板（general/finance/legal）生成确定性 mock 输出。

#### 3.2.2 VerificationAgent（验证 Agent）

**文件**: `backend/app/agents/verification.py`

**职责**: 对每个 finding 进行独立的四维度可信度评估，并检测 findings 间关系。

**设计理念 (v4.0)**:
- 不再"检测矛盾并标记无效"——矛盾可能只是视角差异
- 每个 finding 独立评估，不因存在不一致而降低评分
- 区分三种关系类型：perspective_difference / tension / genuine_contradiction

**评估流程**:
1. 构建 TOON 格式 findings：`@f1|85%|观点摘要`
2. 构建 system prompt (`CREDIBILITY_SYSTEM_PROMPT`) + TOON findings
3. 调用 LLM → 解析 JSON 响应
4. 返回 `(list[CredibilityAssessment], list[FindingRelation])`

**四维度评估模型**:

| 维度 | 权重 | 说明 |
|------|------|------|
| evidence_strength | 0.3 | 论据是否具体、充分、可验证 |
| source_reliability | 0.2 | 分析来源是否可信、权威 |
| reasoning_soundness | 0.3 | 从证据到结论的推理是否严密 |
| data_consistency | 0.2 | 数据是否自洽、与其它 finding 一致 |

加权公式: `overall_credibility = evidence_strength*0.3 + source_reliability*0.2 + reasoning_soundness*0.3 + data_consistency*0.2`

**关系类型**:
- **perspective_difference**: 视角差异——观察维度不同，各自可信
- **tension**: 张力——存在不一致但可通过进一步分析调和
- **genuine_contradiction**: 真实矛盾——逻辑上不可调和

#### 3.2.3 SynthesisAgent（综合 Agent）

**文件**: `backend/app/agents/synthesis.py`

**职责 (v4.0)**: 从报告和评估中提取论证结构图 (LogicGraph) + 为节点生成简洁摘要。

**核心流程**:
1. 构建 credibility map + findings 摘要 + relations 摘要
2. 调用 LLM 提取 `LogicGraph`（节点 + 边）
3. `_generate_node_summaries()` — 为每个节点调用 LLM 生成 1-2 句摘要
4. `_generate_local_summaries()` — 对未获得摘要的节点使用启发式规则截取
5. 返回 `SynthesisResult`

**LogicGraph 节点类型**:
- `question`: 核心问题
- `conclusion`: 综合结论
- `claim`: 分析主张（关联 finding）
- `evidence`: 支持证据

**LogicGraph 边类型**:
- `supports`: 支持关系
- `perspective_difference`: 视角差异
- `tension`: 张力
- `genuine_contradiction`: 真实矛盾

**节点摘要生成** (v4.0 新增):
1. LLM 路径：调用 `NODE_SUMMARY_SYSTEM_PROMPT` 为每个节点生成 ≤30 字中文/≤60 字英文摘要
2. 本地 fallback：从 label 中按标点截断，最多 40 字符
3. 前端画布优先显示摘要而非完整 label

### 3.3 服务层 (services)

#### 3.3.1 LLMClient（LLM 统一客户端）

**文件**: `backend/app/services/llm_client.py`

**职责**: 封装多 LLM 提供商的统一调用接口。

**支持的提供商**:

| Provider | API 格式 | 默认模型 |
|----------|---------|---------|
| chatgpt | OpenAI 兼容 | gpt-4o-mini |
| deepseek | OpenAI 兼容 | deepseek-chat |
| qwen | OpenAI 兼容 | qwen-plus |
| kimi | OpenAI 兼容 | moonshot-v1-8k |
| custom | OpenAI 兼容 | (用户指定) |
| claude | Anthropic 原生 | claude-3-5-sonnet |
| gemini | Gemini 原生 | gemini-1.5-flash |

**路由机制** (`call_llm`):
```python
if provider in {"chatgpt", "deepseek", "qwen", "kimi", "custom"}:
    → _call_openai_compatible()
elif provider == "claude":
    → _call_claude()
elif provider == "gemini":
    → _call_gemini()
else:
    → _call_openai_compatible()  # 降级
```

**黑盒封装 `LLMEngine`**:
- `build_config(api_key, model, base_url)` — 自动推断 provider
- `chat(config, ...)` — 统一聊天接口
- `estimate_call_cost(config, input, output)` — 成本估算

**模型→Provider 自动映射**: 根据模型名称前缀（如 `gpt-4o` → chatgpt, `claude-3` → claude）自动推断，支持防御性修正。

#### 3.3.2 PostProcessor（后处理器）

**文件**: `backend/app/services/post_processor.py`

**职责**: 增强 Markdown 内容，添加审计头和 provenance 标签。

**`with_audit_header()`**: 在 Markdown 内容前添加 YAML 审计头：
```yaml
---
# synapsegrid_audit
session_id: sess_xxx
merkle_root: sha256:xxx
event_count: 15
content_signature: hmac:xxx
---
```

**`enrich()` 函数** — 轻量级纯文本增强（不调用 LLM）:
1. **Provenance 包裹**: 将 `<finding id="f1" confidence="0.85">...</finding>` 替换为 `<provenance finding="f1" credibility="0.86"><finding...>...</provenance>`，关联可信度
2. **可信度摘要**: 在第一个标题后插入彩色可信度表格（GREEN≥80% / YELLOW≥60% / RED）
3. **关系提示**: 在报告末尾添加关系提示（[Perspective] / [WARNING Tension] / [ALERT Contradiction]）

**`_sign_content()`**: 使用 HMAC-SHA256 对 `session_id|merkle_root|event_count|markdown` 进行签名。

#### 3.3.3 TOONSerializer（TOON 序列化器）

**文件**: `backend/app/services/toon_serializer.py`

**职责**: 提供比 JSON 节省 25-40% Token 的紧凑数据格式。

**TOON 格式示例**:
```
findings[3]{id,claim,confidence}:
  f1,"NEV sales +45.2% YoY",0.92
  f2,"CATL battery cost $80/kWh",0.85
  f3,"Regulation tightening in EU",0.73
```

**关键方法**:

| 方法 | 说明 |
|------|------|
| `findings_to_toon()` | finding 列表 → TOON |
| `toon_to_findings()` | TOON → finding 列表 |
| `validation_result_to_toon()` | 验证结果 → TOON |
| `toon_to_validation_result()` | TOON → 验证结果 |
| `compare_size()` | 比较 TOON vs JSON 的字符数 |

**CSV 兼容**: 数据行使用 Python csv 模块解析，自动处理引号包裹和逗号转义。

#### 3.3.4 DiffEngine（增量 Diff 引擎）

**文件**: `backend/app/services/diff_engine.py`

**职责**: 计算两个 finding 列表的增量差异，减少 Agent 间传输的 Token。

**Diff 格式**:
```
diff{
  added[2]:
    findings[2]{id,claim,confidence}:
      f3,"new finding",0.88
  modified[1]:
    findings[1]{id,claim,confidence}:
      f2,"updated claim",0.75
  unchanged[2]: f1,f4
  removed[1]: f5
}
```

**关键方法**:

| 方法 | 说明 |
|------|------|
| `compute_diff(old, new)` | 计算差异 → {added, modified, unchanged, removed} |
| `diff_to_toon(diff)` | Diff → TOON 格式 |
| `toon_to_diff(data)` | TOON → Diff 对象 |
| `apply_diff(base, diff)` | 应用 Diff 到基础列表 |
| `_is_different(a, b)` | 比较 claim/evidence/confidence/source 是否变化 |

**设计用途**: 为 SACP 协议中 Agent 间增量通信做准备，当前版本中 Research → Verification 间使用完整 TOON 传输。

#### 3.3.5 ContextFilter（上下文净化过滤器）

**文件**: `backend/app/services/context_filter.py`

**职责**: 按最小权限原则过滤 Agent 上下文。

**过滤规则**:

| Agent | 可访问事件类型 |
|-------|-------------|
| research | TaskStarted |
| verification | TaskStarted, FindingReported |
| synthesis | FindingValidated, ConflictDetected, FindingRolledBack, SynthesisGenerated |

**Payload 精简**: 过滤后的事件进一步精简 payload，如 Verification Agent 不需要看到其他 finding 的完整 source 字段。

#### 3.3.6 其他服务

| 模块 | 文件 | 职责 |

| TokenCounter | `token_counter.py` | 使用 tiktoken 精确计算 / 字符估算回退 |

### 3.4 基础设施层 (infrastructure)

#### 3.4.1 SessionEventStore（事件存储）

**文件**: `backend/app/infrastructure/event_store.py`

**职责**: 每个会话一个 SQLite 数据库，提供追加式事件持久化。

**数据库表结构**:
```sql
CREATE TABLE events (
    version         INTEGER PRIMARY KEY,
    event_type      TEXT NOT NULL,
    actor           TEXT NOT NULL,
    payload         TEXT NOT NULL,
    metadata        TEXT NOT NULL DEFAULT '{}',
    semantic_digest TEXT NOT NULL DEFAULT '',
    event_hash      TEXT NOT NULL,
    created_at      TEXT NOT NULL
);

CREATE TABLE snapshots (
    version     INTEGER PRIMARY KEY,
    state       TEXT NOT NULL,
    merkle_root TEXT NOT NULL,
    created_at  TEXT NOT NULL
);

CREATE TABLE lineage (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    data        TEXT NOT NULL,
    updated_at  TEXT NOT NULL
);
```

**SQLite 优化 PRAGMA**:
```sql
PRAGMA journal_mode=WAL;         -- 读写不阻塞
PRAGMA wal_autocheckpoint=2000;  -- 减少 I/O 抖动
PRAGMA cache_size=-64000;        -- 64MB 页面缓存
PRAGMA synchronous=NORMAL;       -- WAL 模式下足够安全
PRAGMA busy_timeout=5000;        -- 5 秒锁等待
```

**`append()` 流程**:
1. 获取下一个版本号 `_next_version()` — `SELECT COALESCE(MAX(version), 0) + 1`
2. 设置 version、created_at、event_hash
3. 序列化 actor（Pydantic 模型 → JSON 或 字符串）
4. 写入 events 表并提交
5. 返回已保存的事件

#### 3.4.2 MerkleTree（Merkle 树）

**文件**: `backend/app/infrastructure/merkle.py`

**职责**: 构建事件链的 Merkle 树，提供完整性验证。

**算法**:
1. 每个事件计算 SHA-256 哈希 → `sha256:xxx`
2. 叶子节点两两配对，拼接哈希后再次 SHA-256
3. 奇数节点复制自身配对
4. 递归直到仅剩一个根哈希

**Ed25519 签名** (可选): 使用 PyNaCl 库生成密钥对，对内容签名/验证。

**关键函数**:

| 函数 | 说明 |
|------|------|
| `MerkleTree.append(event_hash)` | 追加事件哈希 |
| `MerkleTree.root_hash` | 获取当前根哈希 |
| `merkle_root(events)` | 批量计算 Merkle 根 |
| `sign_content(content)` | Ed25519 签名 |
| `verify_content(content, sig, pubkey)` | Ed25519 验签 |

#### 3.4.3 LineageGraph（谱系图）

**文件**: `backend/app/infrastructure/lineage_graph.py`

**职责**: 追踪事件之间的依赖关系，可视化事件链。

**结构**: 有向图，节点 = 事件版本（v1/TaskStarted, v2/FindingReported...），边 = 时序关系（next）。

**持久化**: 每次事件追加后同步写入 SQLite `lineage` 表，支持跨进程恢复。

**加载策略**: 优先从 SQLite 恢复，不存在则从事件流重建。

#### 3.4.4 RollbackController（回退控制器）

**文件**: `backend/app/infrastructure/rollback_controller.py`

**职责**: 处理分析要点的回退逻辑。

**execute() 流程**:
1. 验证 finding 存在且未被回退
2. 写入 FindingRolledBack 事件
3. 调用 `ResearchAgent.rebuild()` 基于剩余 findings 重新组织 Markdown
4. 根据 output_mode 分支：
   - **markdown**: `PostProcessor.enrich()` → MarkdownRestored 事件
   - **canvas**: `SynthesisAgent.run()` → SynthesisRestored 事件

#### 3.4.5 CleanupScheduler（清理调度器）

**文件**: `backend/app/infrastructure/cleanup_scheduler.py`

**职责**: 管理会话的延迟清理。

**特性**:
- 持久化调度信息到 SQLite `cleanup_schedule` 表
- 启动时补偿：恢复所有未完成的调度
- 支持取消/重新调度
- 通过 `purge_handler` 回调执行实际清理

## 4. 关键算法

### 4.1 三阶段 Agent 流水线

```
算法：三阶段分析流水线

输入: TaskRequest { prompt, template, output_mode, llm_configs }
输出: TaskResponse { markdown, canvas_schema, audit_metadata }

1. 初始化
   ctx ← session_manager.create_session(user_id)
   写入 TaskStarted 事件

2. 研究阶段 (ResearchAgent)
   raw_markdown, findings ← ResearchAgent.run(prompt, llm_research, template)
   for each finding in findings:
       写入 FindingReported 事件

3. 验证阶段 (VerificationAgent)
   应用模型分层：verification 使用轻量模型
   assessments, relations ← VerificationAgent.run(findings, llm_verification, template)
   估算验证成本
   for each assessment in assessments:
       写入 CredibilityAssessed 事件
   for each relation in relations:
       写入 RelationDetected 事件

4. 综合阶段（双路径分支）
   if output_mode == "markdown":
       enriched_md ← PostProcessor.enrich(raw_markdown, assessments, relations)
       写入 MarkdownGenerated 事件
   else (canvas):
       synth_result ← SynthesisAgent.run(raw_markdown, findings, assessments, relations, llm_synthesis, template)
       写入 SynthesisGenerated 事件（含 logic_graph）

5. 构建响应
   state ← ctx.projection.dump()
   构建 AuditMetadata（含 merkle_root、cost_breakdown 等）
   返回 TaskResponse
```

### 4.2 事件溯源与状态投影

```
算法：事件追加与状态投影

事件追加流程（写路径）:
1. future ← asyncio.create_future()
2. write_queue.put((event, future))
3. saved_event ← await future        // 等待 writer_loop 完成
4. projection.apply(saved_event)      // 投影到内存状态
5. lineage.apply(saved_event)         // 更新谱系图
6. lineage.persist_to_store(conn)     // 持久化谱系图
7. if saved_event.version % 50 == 0:
       event_store.save_snapshot(state, merkle_root)  // 定期快照

Writer Loop（单写者）:
loop:
    event, future ← write_queue.get()
    if event is None: break            // 毒丸信号
    try:
        saved ← event_store.append(event)  // 分配版本号、计算哈希、写入SQLite
        future.set_result(saved)
    except Exception:
        future.set_exception(exc)
    finally:
        write_queue.task_done()

状态投影 (_project):
    if event.event_type == "TaskStarted":
        state["task"] = event.payload
    elif event.event_type == "FindingReported":
        state["findings"][finding["id"]] = finding
    elif event.event_type == "FindingRolledBack":
        state["findings"][id]["rolled_back"] = True
        state["rolled_back_findings"].append(id)
    elif event.event_type == "CredibilityAssessed":
        state["credibility_assessments"][id] = assessment
    ...
```

### 4.3 Merkle 树完整性验证

```
算法：Merkle 树构建与验证

构建 (append):
    leaves.append(event_hash)
    nodes ← leaves.copy()
    while len(nodes) > 1:
        next_level ← []
        for i in 0, 2, 4, ... step 2:
            if i+1 < len(nodes):
                combined ← nodes[i] + nodes[i+1]
            else:
                combined ← nodes[i] + nodes[i]   # 奇数节点复制自身
            hash ← SHA256(combined)
            next_level.append(f"sha256:{hash}")
        nodes ← next_level
    root_hash ← nodes[0]

验证流程:
1. 前端发起 POST /verify/simple { session_id }
2. 后端获取会话的事件列表和 Merkle 根
3. 后端计算最新报告内容的 HMAC 签名
4. 比较计算值 vs 存储值
5. 返回验证结果 { valid: bool, merkle_root, event_count, content_signature }

事件哈希计算:
    raw = f"{event_type}|{actor_json}|{payload_json}|{metadata_json}|{semantic_digest}|{version}"
    event_hash = "sha256:" + SHA256(raw)
```

### 4.4 四维度可信度评估

```
算法：Finding 可信度评估

输入: findings[n] = [{id, claim, confidence, evidence, source}, ...]
输出: assessments[n], relations[m]

1. 构建 TOON 格式
   toon_lines = ["@f1|85%|观点摘要", "@f2|72%|观点摘要", ...]

2. 调用 LLM
   system_prompt ← CREDIBILITY_SYSTEM_PROMPT
   user_prompt ← "请评估以下分析要点的可信度：\n\n" + toon_text
   result ← LLM.chat(system_prompt, user_prompt, temperature=0.3)

3. 解析 JSON 响应
   data ← json.loads(result.content) 或从 ```json``` 提取
   for each item in data["assessments"]:
       assessment ← CredibilityAssessment(
           finding_id         = item.finding_id,
           evidence_strength  = item.evidence_strength,   // 0-1
           source_reliability = item.source_reliability,  // 0-1
           reasoning_soundness= item.reasoning_soundness, // 0-1
           data_consistency   = item.data_consistency,    // 0-1
           overall_credibility = weighted_average(
               evidence_strength*0.3 + source_reliability*0.2
               + reasoning_soundness*0.3 + data_consistency*0.2
           ),
       )

4. 解析关系
   for each item in data["relations"]:
       relation ← FindingRelation(
           between       = item.between,
           relation_type = item.relation_type,  // perspective_difference|tension|genuine_contradiction
           description   = item.description,
       )
```

### 4.5 论证结构图提取

```
算法：LogicGraph 提取 + 节点摘要生成

输入: raw_markdown, findings, assessments, relations, llm_config
输出: SynthesisResult { logic_graph, enriched_markdown, node_map }

1. 构建评估映射
   cred_map[id] ← assessment.overall_credibility

2. 构建 LLM 输入
   user_prompt ← f"## 分析报告摘要\n{raw_markdown[:2000]}\n"
   user_prompt += f"## 分析要点及可信度\n{findings_text}\n"
   user_prompt += f"## 要点间关系\n{relations_text}\n"

3. 调用 LLM 提取
   result ← LLM.chat(LOGIC_GRAPH_SYSTEM_PROMPT, user_prompt, temperature=0.3)
   graph_data ← parse_json(result.content)
   nodes ← [LogicNode(id, type, label, ...) for each in graph_data.nodes]
   edges ← [LogicEdge(from, to, type, ...) for each in graph_data.edges]

4. 生成节点摘要
   a) LLM 摘要路径:
      for each node: 构建 node_descriptions
      result ← LLM.chat(NODE_SUMMARY_SYSTEM_PROMPT, descriptions, temperature=0.2)
      summaries ← parse result
      node.summary ← summaries[node.id]

   b) 本地 fallback (对无摘要节点):
      截取 node.label 的前 40 字符，在标点处断句

5. 构建 node_map (节点 ID → finding ID 映射)
   for each node where node.finding_id:
       node_map[node.id] ← node.finding_id

6. 返回 SynthesisResult
```

### 4.6 TOON 序列化与 Diff 引擎

```
算法：TOON 序列化

序列化 (findings_to_toon):
1. header ← f"findings[{len(findings)}]{{{field1,field2,...}}}:"
2. for each finding:
       values ← []
       for each field:
           if field 不存在: values.append("")
           elif 值含逗号/引号/换行: values.append('"' + escape(值) + '"')
           else: values.append(str(值))
       lines.append("  " + ",".join(values))
3. return "\n".join(lines)

反序列化 (toon_to_findings):
1. 解析 header → 获取字段列表
2. for each data line (以 "  " 开头):
       用 csv.reader 解析
       按 fields 索引构建 finding dict
       类型转换: confidence→float, validated/rolled_back→bool
3. return findings

算法：Diff 计算

compute_diff(old_findings, new_findings):
    old_map ← {id: finding}
    new_map ← {id: finding}

    added    ← new_map.keys - old_map.keys  → 完整 finding
    removed  ← old_map.keys - new_map.keys  → ID 列表
    modified ← {id where id in both AND _is_different(old, new)}
    unchanged← {id where id in both AND NOT _is_different(old, new)}

    return { added, modified, unchanged, removed }

_is_different(old, new):
    return (old.claim != new.claim
         or old.evidence != new.evidence
         or old.confidence != new.confidence
         or old.source != new.source)
```

### 4.7 回退与重新综合

```
算法：Finding 回退

1. 验证
   if finding_id not in state.findings: raise RollbackError
   if state.findings[finding_id].rolled_back: raise RollbackError

2. 写入回退事件
   rollback_ev ← FindingRolledBack(finding_id, reason)
   append_event(ctx, rollback_ev)

3. 获取剩余活跃 findings
   active_findings ← [f for f in all_findings if not f.rolled_back]

4. 无活跃 findings 时:
   生成空报告 → MarkdownRestored 事件

5. 调用 ResearchAgent.rebuild()
   rebuilt_md ← ResearchAgent.rebuild(original_md, finding_id, reason, active_findings, llm)

6. 根据 output_mode 分支:
   markdown:
       enriched_md ← PostProcessor.enrich(rebuilt_md, assessments, relations)
       写入 MarkdownRestored 事件

   canvas:
       synth_result ← SynthesisAgent.run(rebuilt_md, active_findings, ...)
       写入 SynthesisRestored 事件（含 logic_graph）

7. 投影状态自动更新（FindingRolledBack 事件被投影 → marking rolled_back=True）

重新综合（resynthesize）:
   - 与回退类似，但只重新执行 Synthesis
   - 不需要 rebuild markdown（使用现有的 raw_markdown）
   - 主要用于前端检测到已回退 findings 时触发的主动重综合
```

### 5.1 请求-响应流程

```
用户输入 prompt
    ↓
前端 ChatInput → createTask(prompt, llm, agentLlm, template, outputMode)
    ↓
POST /api/v1/tasks { prompt, llm_research, llm_verification, llm_synthesis, template, output_mode }

    ├─ session_manager.create_session()    → SQLite 文件创建 + writer_loop 启动
    ├─ TaskStarted Event                   → append → 投影 → 谱系图
    ├─ ResearchAgent.run()                 → LLM 调用 → (raw_markdown, findings)
    │  └─ FindingReported Events (xN)     → append → 投影 → 谱系图
    ├─ VerificationAgent.run()             → LLM 调用 → (assessments, relations)
    │  ├─ CredibilityAssessed Events (xN) → append → 投影 → 谱系图
    │  └─ RelationDetected Events (xM)    → append → 投影 → 谱系图
    ├─ [分支] output_mode == markdown:
    │  └─ PostProcessor.enrich()          → MarkdownGenerated Event
    └─ [分支] output_mode == canvas:
       └─ SynthesisAgent.run()            → LLM 调用 → SynthesisGenerated Event
    ↓
_build_response()                          → 从投影状态 + 事件流构建 TaskResponse
    ↓
前端接收 TaskResponse → 更新 messages 状态 → 渲染 ChatHistory + AuditPanel/CanvasViewer
```

### 5.2 回退流程

```
用户在审计面板点击"回退" → 确认

    ↓

    ↓
Orchestrator.rollback_finding(ctx, finding_id, reason)
    ├─ 验证 finding 存在且未回退
    ├─ FindingRolledBack Event            → append → 投影
    ├─ ResearchAgent.rebuild()            → LLM 调用 → rebuilt_markdown
    ├─ [分支] markdown: PostProcessor.enrich() → MarkdownRestored Event
    └─ [分支] canvas: SynthesisAgent.run()    → SynthesisRestored Event
    ↓
_build_response() → 更新后的 TaskResponse
    ↓
前端更新 messages 和 activeMsg/auditMsg
```
