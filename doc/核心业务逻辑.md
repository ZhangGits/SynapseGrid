# SynapseGrid 核心业务逻辑

**版本**: 3.1-MVP
**日期**: 2026-06-13

---

## 一、整体架构

```
用户 POST /api/v1/tasks
        │
        ▼
  Orchestrator.run_task()
        │
        ├── 1. 创建 Session（SQLite + NetworkX + WriteQueue）
        │
        ├── 2. Research Agent    ─→ 生成 FindingReported 事件
        ├── 3. Aggregation        ─→ 生成 AggregationCompleted 事件
        ├── 4. Verification Agent ─→ 生成 FindingValidated + ConflictDetected 事件
        │                         ─→ 含预算闸门（budget > 70% 时跳过）
        ├── 5. Synthesis Agent    ─→ 生成 SynthesisGenerated 事件（LLM 输出固化）
        │
        └── 6. 返回 TaskResponse（JSON 复合结构）
```

三步流水线是**串行**执行的：Research → Verification → Synthesis。每个 Agent 生成的事件都通过单写者队列追加到 session 私有的 SQLite，再投影到内存状态。

## 二、Orchestrator（编排器）

**文件**: `backend/app/core/orchestrator.py`

### 2.1 每个 Task 的完整流程

```
run_task(request)
  │
  ├── 解析 per-Agent LLM 配置（r_llm / v_llm / s_llm）
  ├── 写入 TaskStarted 事件
  │
  ├── Phase 1 — Research
  │     └── ResearchAgent.run(prompt, r_llm)
  │           → 返回 FindingReported 事件
  │           → 累加 tokens、扣预算
  │           → 写入 AgentExecution 追踪事件
  │
  ├── 写入 AggregationCompleted 事件
  │
  ├── Phase 2 — Verification（含预算闸门）
  │     │
  │     ├── 如果 budget.would_exceed(500, 200) → 跳过 LLM，走 fallback
  │     └── 否则 → VerificationAgent.run(findings, v_llm)
  │           → 并行验证每条 finding
  │           → 检测冲突
  │           → 写入 AgentExecution 追踪事件
  │
  ├── Phase 3 — Synthesis
  │     └── SynthesisAgent.run(state, s_llm)
  │           → 调用 LLM 生成 Markdown
  │           → 返回 SynthesisGenerated 事件（is_llm_output=True）
  │           → 写入 AgentExecution 追踪事件
  │
  └── _build_response()
        ├── 从 SQLite 读取事件列表
        ├── 从内存投影读取状态
        ├── 计算 Merkle 根 + 内容签名
        ├── 构建 JSON 复合响应
        └── 调度会话清理（默认 3600s 后）
```

### 2.2 关键的永不重新调用 LLM 原则

回退时不重新调用 LLM——从 SQLite 恢复最后的 `SynthesisGenerated` 事件，包装一层回退说明即可。

### 2.3 BudgetTracker

```python
BudgetTracker(budget_usd)
  .would_exceed(tokens_in, tokens_out, model) → bool
  .spend(tokens_in, tokens_out, model)        → cost
  .remaining                                  → float
```

## 三、三个 Agent

### 3.1 Research Agent — 任务分解

**文件**: `backend/app/agents/research.py`

**输入**: 用户 prompt + LLMConfig
**输出**: `FindingReported` 事件列表（f1, f2, f3...）

**核心逻辑**:
1. 如果有 API key → 调用 LLM 提取 findings
2. 如果没有 → fallback（生成 2 条通用占位 finding）
3. 每条 finding 是一个 `{claim, source, confidence}` 对象

**关键特性**:
- `response_format={"type":"json_object"}` 约束 LLM 输出
- **Finding 数量自适应**: prompt < 50 字符→2-3 条，50-200→3-5 条，> 200→4-6 条
- **大任务拆分**: prompt > 200 字符时，拆为 2 份并行调用（`asyncio.gather`），合并去重后最多 6 条
- Claude/Gemini 走 httpx 直调，OpenAI-compatible 走 SDK
- 失败重试 2 次
- JSON 解析有 5 层 fallback（json_object→数组→正则→逐行→最后手段）

### 3.2 Verification Agent — 验证 + 冲突检测

**文件**: `backend/app/agents/verification.py`

**输入**: findings 列表 + LLMConfig
**输出**: `FindingValidated` + `ConflictDetected` 事件

**核心逻辑**:
1. Phase 1（**并行**）: 每条 finding 独立调用 LLM 验证，调整置信度
2. Phase 2: 批量 LLM 调用检测所有 finding 之间的矛盾对

**关键特性**:
- `asyncio.gather` 并行验证所有 finding
- 验证失败的单条不阻塞其余
- 冲突检测仅在 findings ≥ 2 时执行
- 同样支持 4 种 provider，`response_format` 约束

### 3.3 Synthesis Agent — 最终回答生成

**文件**: `backend/app/agents/synthesis.py`

**输入**: 用户 prompt + 验证后的 findings + 冲突列表 + LLMConfig
**输出**: `SynthesisGenerated` 事件（`is_llm_output=True`）

**核心逻辑**:
1. 将 findings 压缩为 ID+摘要（blackboard 协议，节约 ~70% tokens）
2. 调用 LLM 生成最终 Markdown 报告
3. LLM 输出作为不可变事件固化到 SQLite

**关键特性**:
- System prompt 强制结构化输出: `## 结论` → `## 关键发现` → `## 风险评估` → `## 方法论`
- 同样支持多 provider + fallback

## 四、会话上下文管理器（Session）

**文件**: `backend/app/core/session_manager.py`

### 4.1 会话创建

```
create_session(user_id)
  → 创建 SQLite 文件 (/tmp/synapse_sessions/{session_id}.db)
  → 创建 NetworkX 血缘图
  → 创建单写者队列 (asyncio.Queue)
  → 启动 writer_loop
  → 返回 SessionContext
```

### 4.2 事件追加（先持久化后投影）

```
append_event(ctx, event)
  1. 入队 → write_queue.put((event, future))
  2. 等待写入完成 → await future
  3. 投影到内存 → ctx.projection.apply(saved)
  4. 更新血缘图 → ctx.lineage.apply(saved)
  5. 序列化图谱回 SQLite → ctx.lineage.persist_to_store()
  6. 每 50 个事件 → 打快照到 snapshots 表
```

### 4.3 清理

- 任务结束后调度 `cleanup_scheduler`（默认延迟由 `cleanup_delay_seconds` 决定，最小 3600s）
- 清理时关闭 writer loop、删除 SQLite 文件（含 WAL/SHM）
- 启动时补偿清理上次崩溃遗留

## 五、事件存储与状态投影

### 5.1 Event Store

**文件**: `backend/app/infrastructure/event_store.py`

- 每 session 一个 SQLite 文件，WAL 模式
- 事件表: event_id, event_type, aggregate_id, version, timestamp, actor_json, payload_json, metadata_json, previous_hash, event_hash, signature, is_llm_output
- 哈希链: `previous_hash` 指向前一事件，`event_hash = SHA256(canonical JSON of event)`
- 快照表: version, state_blob (JSON), merkle_root
- 每次 `append()` 计算哈希并签名

### 5.2 State Projection

**文件**: `backend/app/core/state_machine.py`

```
事件类型 → 状态投影:
  TaskStarted          → state["task"]
  FindingReported      → state["findings"][id]
  FindingValidated     → state["findings"][id]["validated"] = True
  FindingRolledBack    → state["findings"][id]["rolled_back"] = True
  AggregationCompleted → state["aggregation"]
  ConflictDetected     → state["conflicts"][id]
  SynthesisGenerated   → state["synthesis"]
  SynthesisRestored    → state["synthesis_restored"]
  AgentExecution       → state["agent_traces"].append(payload)
```

## 六、回退控制器

**文件**: `backend/app/infrastructure/rollback_controller.py`

```
rollback(finding_id, reason)
  1. 验证 finding 在血缘图中存在
  2. 计算下游影响
  3. 写入 FindingRolledBack 补偿事件
  4. 从 SQLite 恢复 SynthesisGenerated 事件
  5. 包装回退说明 Markdown
  6. 写入 SynthesisRestored 事件
  7. 返回新版本号
```

**关键字原则**: **LLM 输出作为不可变事件固化**——回退时重放恢复，永不重新调用 LLM。

## 八、前后端交互

### 请求

```json
POST /api/v1/tasks
{
  "user_id": "user001",
  "prompt": "...",
  "budget_usd": 0.08,
  "llm": { "provider": "deepseek", "model": "deepseek-chat", "api_key": "sk-..." },
  "llm_research": null,
  "llm_verification": null,
  "llm_synthesis": null,
  "conversation_history": "Q1: ...\nA1: ..."
}
```

per-Agent LLM 字段为空时，各 Agent 自动回退到顶层 `llm` 配置。

### 响应

JSON 复合结构，同时包含人类可读的 Markdown 和机器可读的审计元数据。前端左栏渲染 Markdown（带打字机效果），右栏渲染审计面板（含折线图、执行追踪、回退控件）。

### 回退

```
POST /api/v1/sessions/{session_id}/rollback
{ "finding_id": "f2", "reason": "不可靠" }
```

返回更新后的 TaskResponse，Markdown 包含回退说明但保留原始 LLM 回答。

### 验证

```
POST /api/v1/sessions/{session_id}/verify
→ { session_id, merkle_root, content_signature, event_count }
```
