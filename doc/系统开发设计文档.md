## 1. 系统概述

### 1.1 项目定位

SynapseGrid 是一个**可审计的事件溯源 AI Agent 系统**，采用三阶段 Agent 流水线（Research → Verification → Synthesis）执行分析任务。核心创新在于：

- **事件溯源（Event Sourcing）**：所有 Agent 操作均记录为不可变事件，支持完整审计追溯
- **Merkle 树完整性验证**：事件链通过 Merkle 树保证数据完整性
- **可回退机制**：支持对单个分析要点（Finding）进行回退，并自动重新综合报告
- **双输出模式**：支持 Markdown 和 Canvas（结构化可视化）两种输出格式

### 1.2 技术栈

| 层级 | 技术 | 用途 |
|------|------|------|
| 后端框架 | FastAPI (Python 3.11+) | REST API 服务 |
| 数据存储 | SQLite (WAL 模式) | 事件存储、快照、谱系图 |
| LLM 客户端 | httpx (异步 HTTP) | 统一调用 OpenAI/Claude/DeepSeek 等 |
| 前端框架 | React 18 + TypeScript | SPA 用户界面 |
| 构建工具 | Vite | 前端构建与开发服务器 |
| Markdown 渲染 | react-markdown | 报告展示 |
| 容器化 | Docker + Docker Compose | 部署运行 |

### 1.3 核心概念

| 概念 | 说明 |
|------|------|
| **Event（事件）** | 不可变的日志条目，记录 Agent 的每次操作 |
| **Session（会话）** | 一次分析任务的完整生命周期，包含事件流 |
| **Finding（分析要点）** | Research Agent 生成的单个分析结论 |
| **Conflict（矛盾）** | Verification Agent 检测到的 finding 间逻辑矛盾 |
| **Merkle Root（Merkle 根）** | 事件链的完整性校验和 |
| **Canvas Schema（画布模式）** | 结构化的可视化数据格式 |

### 2.2 分层架构

系统采用经典的三层架构：

#### 2.2.1 API 路由层 (`app/api/v1/`)

- **tasks.py**: 任务 CRUD 端点（创建、查询、回退、反馈、关闭、重新综合）
- **verify.py**: 完整性验证端点（内容验证、事件重放、公钥获取）

#### 2.2.2 业务逻辑层 (`app/core/` + `app/agents/`)

- **verification.py**: 验证 Agent — finding 验证与矛盾检测
- **synthesis.py**: 综合 Agent — 报告生成

#### 2.2.3 基础设施层 (`app/infrastructure/`)

- **event_store.py**: SQLite 事件持久化

- **lineage_graph.py**: 事件谱系图
- **rollback_controller.py**: 回退控制逻辑
- **cleanup_scheduler.py**: 会话清理调度

## 3. 后端模块设计

### 3.1 核心数据模型 (`app/schemas/`)

#### 3.1.1 Event（事件）— `machine_wire.py`

事件是系统的核心不可变数据单元：

```python
class Event(BaseModel):
    version: int = 0           # 自增版本号（由存储分配）
    event_type: str = ""       # 事件类型
    actor: LlmConfig | str     # 执行者（LLM 配置或字符串）
    payload: dict              # 事件负载
    metadata: dict             # 元数据（用户ID、成本等）
    event_hash: str = ""       # SHA-256 哈希
    created_at: str = ""       # ISO 格式 UTC 时间
```

**事件类型枚举**：

| 事件类型 | 触发阶段 | 说明 |
|----------|----------|------|
| `TaskStarted` | Research | 任务开始，记录 prompt 和配置 |
| `FindingReported` | Research | 生成一个分析要点 |
| `FindingValidated` | Verification | 验证通过一个 finding |
| `ConflictDetected` | Verification | 检测到矛盾 |
| `SynthesisGenerated` | Synthesis | 生成综合报告 |
| `SynthesisRestored` | Rollback | 回退后重新生成报告 |
| `FindingRolledBack` | Rollback | 标记 finding 已回退 |
| `CanvasUpdated` | Feedback | 画布模式更新 |
| `AgentExecution` | 任意 | Agent 执行追踪 |

### 3.2 会话管理 (`session_manager.py`)

#### 3.2.1 会话生命周期

```
创建会话 → 执行任务 → 追加事件 → 关闭会话 → 延迟清理 → 删除文件
```

- **创建**: 生成唯一 `session_id`（格式: `sess_{user_id}_{uuid}_{timestamp}`）
- **存储**: 每个会话一个 SQLite 数据库文件（`/tmp/synapse_sessions/{session_id}.db`）
- **写入**: 单写者队列模式（`asyncio.Queue`），确保串行化写入
- **清理**: 关闭后默认延迟 300 秒清理，支持持久化调度和启动补偿

#### 3.2.2 单写者队列模式

```
append_event(ctx, event)
    ↓
write_queue.put((event, future))  ← 入队
    ↓
_writer_loop(ctx)                 ← 唯一写者协程
    ↓
event_store.append(event)         ← 写入 SQLite
    ↓
future.set_result(saved)          ← 通知调用方
    ↓
projection.apply(saved)           ← 更新内存投影
lineage.apply(saved)              ← 更新谱系图
```

**设计理由**：避免多协程并发写入 SQLite 导致的锁竞争和死锁问题。

### 3.3 状态投影 (`state_machine.py`)

#### 3.3.1 投影机制

事件溯源的核心模式：从事件流重建应用状态。

```
事件流: [TaskStarted, FindingReported, FindingValidated, ...]
                    ↓
投影函数 _project(event, state) 逐个应用
                    ↓
当前状态: { task, findings, conflicts, synthesis, ... }
```

#### 3.3.2 快照加速

每 50 个事件保存一次状态快照，历史状态查询时从最近快照开始重放：

```
get_state_at(target_version)
    ↓
snapshot = get_latest_snapshot(max_version=target_version)
    ↓
tail_events = get_events(from_version=snapshot.version + 1, to_version=target_version)
    ↓
对 tail_events 逐个应用 _project()
```

### 3.4 Agent 流水线 (`agents/`)

#### 3.4.1 Research Agent（研究 Agent）

**职责**：将用户问题拆解为多个分析要点（Finding）

**流程**：
1. 检查 API key，若无则使用 mock 输出
2. 构建系统提示词 + 模板上下文 + 用户 prompt
3. 调用 LLM，要求返回 JSON 格式的 findings 列表
4. 解析 JSON 响应，转换为标准 finding 格式
5. 每个 finding 包含：id, claim, evidence, confidence, source, agent, model, cost_usd

**系统提示词要点**：
- 要求以 JSON 数组格式返回
- 每个 finding 需有唯一 id（f1, f2, f3...）
- confidence 为 0-1 浮点数
- 生成 3-5 个分析要点

**Mock 模式**：根据模板类型（general/finance/legal）生成不同的 mock findings

#### 3.4.2 Verification Agent（验证 Agent）

**职责**：验证 findings 并检测逻辑矛盾

**流程**：
1. 构建 findings 摘要（id + claim + confidence）
2. 调用 LLM 进行验证
3. 解析返回的 JSON：`{ validated_ids, conflicts }`
4. 返回 `(validated_ids, conflicts)` 元组

**矛盾类型**：
- `high`: 严重矛盾（完全对立）
- `medium`: 中等矛盾（部分不一致）
- `low`: 轻微不一致

#### 3.4.3 Synthesis Agent（综合 Agent）

**职责**：将验证后的 findings 综合为最终报告

**流程**：
1. 构建 findings 摘要和矛盾摘要
2. 调用 LLM 生成完整报告
3. 根据 output_mode 决定是否生成 canvas_schema

**输出模式**：
- **Markdown 模式**：返回纯 Markdown 文本
- **Canvas 模式**：额外生成结构化 CanvasSchema，包含：
  - 标题组件（markdown）
  - KPI 概览卡片（finding 数、平均置信度、矛盾数）
  - 完整报告（markdown）
  - 各 finding 详情（finding_box）
  - 矛盾详情（markdown）

#### 3.6.2 SQLite 性能优化

```sql
PRAGMA journal_mode=WAL;      -- WAL 模式，读写不互斥
PRAGMA synchronous=NORMAL;    -- 平衡性能与安全性
PRAGMA busy_timeout=5000;     -- 忙等待超时
PRAGMA foreign_keys=ON;       -- 外键约束
```

### 3.7 Merkle 树 (`merkle.py`)

#### 3.7.1 树结构

```
        root = sha256(h1 + h2)
       /                    \
 h1 = sha256(l1 + l2)    h2 = sha256(l3 + l4)
   /        \              /        \
 l1         l2           l3         l4
(event1)  (event2)     (event3)   (event4)
```

- 叶子节点：事件哈希（`sha256:{hex}`）
- 内部节点：子节点哈希拼接后再次哈希
- 奇数节点：复制自身配对

#### 3.7.2 事件哈希计算

```python
def compute_hash(self):
    raw = f"{event_type}|{actor}|{json(payload)}|{json(metadata)}|{version}"
    return f"sha256:{sha256(raw.encode()).hexdigest()}"
```

#### 3.7.3 Ed25519 签名（可选）

依赖 PyNaCl 库，用于第三方审计场景：
- `sign_content(content)`: 使用私钥签名
- `verify_content(content, signature, public_key)`: 验证签名
- `get_public_key_hex()`: 获取公钥

### 3.8 回退控制器 (`rollback_controller.py`)

#### 3.8.1 回退流程

```
rollback_finding(ctx, finding_id, reason)
    │
    ├─ 1. 验证 finding 存在且未回退
    ├─ 2. 写入 FindingRolledBack 事件
    ├─ 3. 从原始 synthesis 中移除被回退 finding 的内容
    ├─ 4. 添加 Rollback Notice 标记
    └─ 5. 写入 SynthesisRestored 事件
```

#### 3.8.2 内容过滤算法

```python
# 从 markdown 中移除被回退 finding 的相关内容
lines = original_markdown.split("\n")
filtered_lines = []
skip_block = False
for line in lines:
    # 精确匹配 finding 标题行
    is_finding_header = (
        line.strip().startswith(f"**{excluded_id}**")
        or line.strip().startswith(f"### {excluded_id}")
        or line.strip().startswith(f"{excluded_id}.")
    )
    if is_finding_header:
        skip_block = True
        continue
    # 遇到下一个 finding 标题或空行后结束跳过
    if skip_block and (line.strip().startswith("**f") 
                       or line.strip().startswith("### f") 
                       or line.strip() == ""):
        skip_block = False
    if not skip_block:
        filtered_lines.append(line)
```

## 5. 关键算法与数据结构

### 5.1 事件溯源模式

```
写入路径:  append → SQLite (持久化) → Projection (内存) → Lineage (内存+SQLite)
查询路径:  SQLite (事件) → Projection.rebuild() (重建状态)
恢复路径:  SQLite (快照) → 重放增量事件 → 重建完整状态
```

### 5.2 Merkle 树算法

```
构建:
  leaves = [hash1, hash2, hash3, ...]
  while len(nodes) > 1:
    for i in range(0, len(nodes), 2):
      if i+1 < len(nodes):
        combined = nodes[i] + nodes[i+1]
      else:
        combined = nodes[i] + nodes[i]  # 奇数节点复制
      h = sha256(combined)
      next_level.append(h)
    nodes = next_level
  root = nodes[0]

验证:
  重新计算 Merkle 根 → 与存储的根比较 → 一致则完整性通过
```

### 5.3 内容签名算法

```
HMAC-SHA256:
  key = SYNAPSEGRID_HMAC_KEY (环境变量)
  msg = f"{session_id}|{merkle_root}|{event_count}|{markdown}"
  signature = "hmac:" + hmac.new(key, msg, sha256).hexdigest()
```

### 5.4 成本估算算法

```
估算 Token:
  tokens = cost_usd / (model_pricing[model] / 1000)

估算费用:
  cost = (input_tokens × input_price + output_tokens × output_price) / 1000

Verification 成本:
  input_tokens = finding_count × 200
  output_tokens = 300

Synthesis 成本:
  input_tokens = finding_count × 300
  output_tokens = 2000
```

### 5.5 Token 计数

优先使用 tiktoken 库精确计算，回退到字符估算：

```python
def count_tokens(text, model):
    if tiktoken available:
        encoding = tiktoken.encoding_for_model(model)
        return len(encoding.encode(text))
    else:
        return max(1, len(text) // 3)  # 中英文混合约 3 字符/Token
```

## 6. 数据流分析

### 6.1 完整任务数据流

```
用户输入 prompt
    │
    ▼
前端 POST /api/v1/tasks
    │
    ▼
Orchestrator.run_task(request)
    │
    ├─ SessionManager.create_session()
    │   └─ 创建 SQLite 数据库 + 启动 writer_loop
    │
    ├─ Event(TaskStarted) → append_event → SQLite + Projection + Lineage
    │
    ├─ ResearchAgent.run(prompt, llm, template)
    │   ├─ call_llm() → LLM API → JSON findings
    │   └─ Event(FindingReported) × N → append_event × N
    │
    ├─ VerificationAgent.run(findings, llm, template)
    │   ├─ call_llm() → LLM API → {validated_ids, conflicts}
    │   ├─ Event(FindingValidated) × N → append_event × N
    │   └─ Event(ConflictDetected) × N → append_event × N
    │
    ├─ SynthesisAgent.run(findings, validated, conflicts, llm, template, mode)
    │   ├─ call_llm() → LLM API → markdown (+ canvas_schema)
    │   └─ Event(SynthesisGenerated) → append_event
    │
    └─ _build_response(ctx, request)
        ├─ 从 Projection 提取 findings, conflicts
        ├─ 从 Events 提取 cost_records
        ├─ with_audit_header() → 计算 HMAC 签名
        └─ 返回 TaskResponse
            │
            ▼
前端接收 TaskResponse
    ├─ 更新消息列表
    ├─ 侧边栏显示审计面板 / 画布模式
    └─ 用户可查看报告或回退 finding
```

### 6.2 回退数据流

```
用户点击"回退"按钮
    │
    ▼
前端 POST /api/v1/tasks/{session_id}/rollback
    │
    ▼
Orchestrator.rollback_finding(ctx, finding_id, reason)
    │
    ├─ RollbackController.execute(ctx, finding_id, reason)
    │   ├─ 验证 finding 存在且未回退
    │   ├─ Event(FindingRolledBack) → append_event
    │   ├─ _rebuild_synthesis() → 过滤内容 + 添加通知
    │   └─ Event(SynthesisRestored) → append_event
    │
    └─ _build_response(ctx, request) → 返回更新后的 TaskResponse
        │
        ▼
前端更新消息 + 审计面板 + 画布
```

### 6.3 重新综合数据流

```
用户确认重新综合
    │
    ▼
前端 POST /api/v1/tasks/{session_id}/resynthesize
    │
    ▼
Orchestrator.resynthesize(ctx)
    │
    ├─ 从 Projection 获取未回退的 findings
    ├─ 从 Events 获取 synthesis 的 LLM 配置
    ├─ SynthesisAgent.run(active_findings, ...)
    │   └─ call_llm() → 生成新报告
    ├─ Event(SynthesisGenerated) → append_event
    └─ _build_response() → 返回更新后的 TaskResponse
```

## 7. 安全设计

### 7.1 当前安全措施

| 措施 | 实现 | 说明 |
|------|------|------|
| CORS 配置 | `allow_origins=["*"]` | 开发环境允许所有来源 |
| HMAC 内容签名 | `post_processor.py` | 防止报告内容被篡改 |
| Ed25519 签名 | `merkle.py`（可选） | 第三方审计签名 |
| API Key 保护 | 前端密码输入框 | 不持久化到后端 |
| SQLite WAL 模式 | `PRAGMA journal_mode=WAL` | 防止数据损坏 |

### 7.2 安全警告

> ⚠️ **生产环境必须修改以下配置：**

1. **CORS 配置**：`allow_origins=["*"]` 应改为具体的前端域名
2. **HMAC 密钥**：默认密钥 `dev-only-change-me-must-be-32-bytes` 必须通过环境变量 `SYNAPSEGRID_HMAC_KEY` 设置
3. **API Key 传输**：当前通过 HTTP 明文传输，生产环境应使用 HTTPS
4. **会话存储路径**：`/tmp/synapse_sessions/` 在容器重启后丢失，生产环境应挂载持久卷
