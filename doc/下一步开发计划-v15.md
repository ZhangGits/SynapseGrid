| 问题 | 现实影响 |
|------|----------|
| **LLM 调用无重试** | 网络抖动一次（概率约 0.5-2%）→ 用户等待 2 分钟后看到"任务失败"，需手动重来。对非技术用户是致命体验。 |

| **无 Agent 层测试** | 修改 Research Prompt 后不知道是否打破了解析逻辑——只能手动验证。 |

## Phase 1: 可靠性加固（Week 1）

### 1.1 LLM 调用重试机制（2 天）

**现实意义**:

> **问题场景**: 用户提交了一个分析任务，等了 30 秒——Research Agent 调 OpenAI 时网络闪断。当前行为：整个任务直接失败，前端显示"Task failed"。用户必须重新输入 prompt、重新等待。如果这是第 3 次尝试，用户大概率放弃使用。
>
> **改造后**: 网络闪断 → 自动等待 1 秒重试 → 成功。用户无感知。只有连续 4 次都失败才报错。这对使用国外 API（跨境网络波动频繁）的国内用户尤为重要。

**技术方案**: 在 `call_llm()` 三个底层调用函数（`_call_openai_compatible`、`_call_claude`、`_call_gemini`）中包裹指数退避重试：第 1 次等 1s、第 2 次等 2s、第 3 次等 4s。仅重试 5xx/超时/连接错误，4xx（如 401 认证失败、429 限流）不重试。

**验收**: 模拟断网 → 自动重试后成功；模拟 API key 错误 → 不重试直接报错。

---

### 1.2 Mock 模式透明化（2 天）

**现实意义**:

> **问题场景**: 用户在金融分析场景下配了一个 DeepSeek API key，但 key 写错了一个字母。系统静默降级到 mock 输出，给了一条"新能源汽车销量增长 12%"的分析。用户基于这条假数据做了投资决策。这是系统最危险的行为——**在用户不知情的情况下输出假数据**。
>
> **改造后**: 
> 1. 后端写入 `AgentDegraded` 事件（"Research Agent 因 API key 无效降级为模拟输出"）
> 2. 前端审计面板顶部显示**醒目的黄色警告横幅**："⚠️ 注意：Research Agent 因 API 错误降级为模拟输出，以下数据并非基于实际 AI 分析"
> 3. 用户一眼就知道这份报告不可信
>
> 这对于金融分析、法律分析等**高可靠性要求的场景**是底线需求。

**技术方案**: 
- 三个 Agent 增加 `_degraded: bool` + `_degrade_reason: str` 属性
- Orchestrator 检测降级后写入 `AgentDegraded` 事件
- 前端 AuditPanel 检测事件流中的 `AgentDegraded` → 渲染警告横幅

**验收**: 无 API key → 前端显示黄色警告；正常模式 → 无警告。

---

### 1.3 统一定价表引用（1 天）

**现实意义**:

> **问题场景**: 未来 DeepSeek 调整了定价（如 deepseek-chat 从 $0.00014 涨到 $0.0002），如果只更新了 `cost_tracker.py` 而漏掉了 `orchestrator.py` 中的硬编码定价，那么审计面板显示的成本和实际扣费不一致。用户发现"系统说花了 $0.05，实际 API 账单是 $0.08"，信任崩塌。
>
> **改造后**: 全局只有一处定价定义（`cost_tracker.MODEL_PRICING`），修改一次全局生效。成本数据始终准确。

**技术方案**: 确认 `orchestrator.py` 中 `_get_avg_price()` 引用 `cost_tracker.MODEL_PRICING`，删除可能的残留硬编码。

## Phase 2: 可观测性补全（Week 2）

### 2.1 SACP 协议流水线集成（2 天）

**现实意义**:

> **问题场景**: 用户想了解"Verification Agent 收到了什么样的数据来做评估？"当前审计面板只能看到每个 Agent 的输出事件（`FindingReported`、`CredibilityAssessed`），但看不到 Agent 之间传递了什么。如果 Verification 的可信度评估结果很奇怪（比如所有 finding 都是 0.5），用户无法追溯是输入数据的问题还是 LLM 的问题。
>
> **改造后**: 每次 Agent 间通信都记录一条 `AgentMessage` 事件，包含：
> - `from_agent → to_agent`（谁传给了谁）
> - `semantic_digest`（一句话摘要："Research 完成，生成 4 个分析要点"）
> - `payload` 完整数据 + SHA-256 校验和（可展开查看）
> - `payload.format`（toon/json/diff）
>
> 前端审计面板新增"Agent 消息流"视图：一条时间线展示 Agent 间的完整通信链路。这使审计面板从"结果展示"升级为**全链路可追溯**。

**技术方案**: 
- Orchestrator 在 Research → Verification 和 Verification → Synthesis 之间构建 `AgentMessage` 并写入事件流
- 新增 `AgentMessage` 事件类型到 `state_machine._project()`
- 前端 AuditPanel 增加 AgentMessage 时间线展示

**验收**: 每个 Agent 间通信都有 `AgentMessage` 事件；前端可展开查看完整 payload；校验和验证通过。

---

### 2.2 CostTracker 流水线集成（1 天）

**现实意义**:

> **问题场景**: 用户使用 gpt-4o（$0.01/1K output tokens）做了一次分析，但不知道实际花了多少钱。审计面板显示的是**估算**成本（Verification 和 Synthesis 阶段用固定公式估算），可能与实际 API 扣费有 10-20% 误差。对于需要精确核算成本的企业用户，这是不可接受的。
>
> **改造后**: `CostTracker` 在每次 LLM 调用后实时记录实际 Token 消耗和费用。审计面板可以展示**精确到 0.0001 美元**的成本明细。用户知道自己每一分钱花在了哪个 Agent 的哪个模型上。

**技术方案**: Orchestrator 在每次 `call_llm()` 返回后调用 `self.cost_tracker.record_call()`，记录 session_id、agent、model、tokens_input、tokens_output。

**验收**: `CostTracker.get_total_cost()` 返回精确总费用；与 API 账单一致。

---

### 2.3 ContextFilter 流水线激活（2 天）

**现实意义**:

> **问题场景**: Synthesis Agent 在生成综合报告时，Orchestrator 可能把整个事件流（包括 Research 的原始 LLM 响应、TaskStarted 的完整 prompt）都塞进上下文。这导致 Synthesis 的输入 Token 远大于实际需要。以 4 个 findings 为例，全量上下文约 3000 tokens，过滤后只需要约 1800 tokens——节省 40%。
>
> **改造后**: 
> - Research: 只接收 TaskStarted（不含其他 finding）
> - Verification: 只接收 TaskStarted + FindingReported
> - Synthesis: 只接收 CredibilityAssessed + RelationDetected
>
> 每个 Agent 的上下文精简 30-60%，**直接节省 API 费用**。按每月 1000 次分析任务、每次平均 0.05 美元计算，全年节省约 $120-240。

**技术方案**: 在流水线中集成 `ContextFilter.filter_for_agent()`，日志记录过滤前后的 Token 差异。

**验收**: 日志显示过滤节省的 Token 数；合成阶段 Token 减少 30%+。

## Phase 3: Token 成本优化（Week 3）

### 3.1 TOON 格式全链路启用（2 天）

**现实意义**:

> **问题场景**: Research Agent 生成 4 个 findings，每个包含 200 字符的 claim 和 500 字符的 evidence。用 JSON 传输给 Verification Agent 约 3500 字符；用 TOON 格式约 2200 字符——**节省 37%**。不要小看这几百个 Token：Verification Agent 和 Synthesis Agent 都会把 findings 放入 Prompt 中，节省的是**输入 Token**（通常占 LLM 调用成本的 60-80%）。
>
> **改造后**: Agent 间通信默认使用 TOON 格式。每个任务的 Agent 间通信 Token 减少 25-40%。按每天 50 次分析计算，每月节省约 $15-30。

**技术方案**: 
- `AgentMessage.payload` 从 `from_json()` 改为 `from_toon()`
- 增加 TOON 解析失败时自动回退 JSON 的降级逻辑
- 基准测试脚本对比 JSON vs TOON 字符数

**验收**: TOON 格式比 JSON 节省 25%+ 字符（基准测试验证）；降级逻辑正常。

---

### 3.2 Diff 增量传输启用（2 天）

**现实意义**:

> **问题场景**: 用户进行多轮对话分析——第 1 轮"分析新能源市场"生成 4 个 findings，第 2 轮追问"重点关注电池技术"——Research Agent 重新生成 5 个 findings，其中 3 个和上一轮几乎一样。当前行为：把 5 个 findings 完整传给 Verification Agent（约 4000 tokens）。实际只需要传 2 个新增的 + 标记 3 个未变化的——**节省 60% 以上**。
>
> **改造后**: 多轮对话中使用 Diff 传输。首轮（无历史）仍使用完整列表。这对于**对话式分析**场景（用户逐步深入追问）效果最显著。

**技术方案**: 
- Orchestrator 检测 `conversation_history` 存在时，从上一轮投影中获取 findings 计算 Diff
- `AgentMessage.payload` 使用 `from_diff()` 格式
- Diff 解析失败时回退到完整 TOON 列表

**验收**: 多轮对话中 Diff 比完整列表节省 60%+ 字符（当 80% findings 未变时）。

---

### 3.3 Token 成本基准测试（1 天）

**现实意义**:

> **问题场景**: 我们在文档里说"TOON 节省 25-40%""Diff 节省 60%+"，但这些都是理论值。实际效果取决于 findings 的数量、文本长度、变化比例。没有基准测试数据，这些"节省"只是空话。用户问"到底能省多少？"时我们回答不出来。
>
> **改造后**: 有一套可重复运行的基准测试，对 3 种场景（单轮通用、多轮对话、金融分析）实测 Token 消耗。生成 `BENCHMARK.md` 报告，用具体数字说话：
> ```
> | 场景       | JSON (chars) | TOON (chars) | 节省   |
> |------------|-------------|-------------|--------|
> | 3 findings | 2,340       | 1,612       | 31.1%  |
> | 5 findings | 4,120       | 2,730       | 33.7%  |
> ```

**技术方案**: 新建 `backend/benchmarks/token_cost.py`；使用 tiktoken 精确计算 Token 数；输出 Markdown 表格。

**验收**: 基准测试可 CI 运行；生成 `BENCHMARK.md`；包含 JSON vs TOON vs Diff 三列对比。

## Phase 4: 测试补全（Week 4）

### 4.1-4.3 Agent 层单元测试（3 天）

**现实意义**:

> **问题场景 1**: 你修改了 Research Agent 的 `FINDINGS_MARKDOWN_PROMPT`，加了一句"请用中文回答"。部署上线 2 小时后用户反馈"分析要点全是英文"。为什么？因为改动后 `parse_finding_tags()` 的正则不再匹配某些边缘格式，触发了 JSON fallback，而 JSON fallback 解析出来的 findings 少了 `evidence` 字段。没有单元测试，你只能靠用户投诉来发现问题。
>
> **问题场景 2**: 你想把 Verification Agent 的可信度评估从四维度改成五维度（加一个 `timeliness` 维度）。你改了 Prompt 和 `CredibilityAssessment` 模型，看起来没问题。但 `state_machine._project()` 中处理 `CredibilityAssessed` 事件的代码假设只有 4 个维度，结果前端审计面板的 Findings 卡片崩溃了。没有测试，你发现这个问题的方式是——前端白屏。
>
> **改造后**: 每次修改 Agent 代码后运行 `pytest backend/tests/test_research_agent.py`，30 秒内验证所有解析逻辑、mock 输出、rebuild 行为没有退化。**测试是重构的安全网**：有了它，你才敢改代码。

**测试清单**:
- **Research Agent (7 用例)**: mock 输出（general/finance/legal）、`<finding>` 标签解析、JSON fallback 解析、rebuild 文本重建、回退通知插入
- **Verification Agent (5 用例)**: mock 评估（general/finance/legal）、可信度范围校验 [0.3, 0.95]、无 API key 降级
- **Synthesis Agent (5 用例)**: mock 节点/边数量校验、节点类型分布、本地摘要截断、node_map 映射

### 4.4-4.5 集成测试（2 天）

**现实意义**:

> **问题场景**: 你在 `orchestrator.py` 中把 Markdown 模式的处理逻辑从 `PostProcessor.enrich()` 改成了直接返回原始 markdown（为了"简化代码"）。你觉得没问题，因为"反正前端渲染 markdown 不需要 enrich"。但你不知道的是：`enrich()` 函数还负责注入 `<provenance>` 标签，而前端 `ProvenanceRenderer` 依赖这些标签来展示 finding 详情弹窗。生产环境上线后，Canvas 点击 finding 节点不再弹出详情。这是一个**跨层集成 Bug**——单独测 Agent 或单独测前端都发现不了。
>
> **改造后**: 5 个集成测试覆盖完整数据流：
> 1. Markdown 全流程：创建任务 → 验证事件类型序列 → 验证 TaskResponse
> 2. Canvas 全流程：验证 LogicGraph 非空、节点数 = finding 数
> 3-4. 回退流程（两种模式）：验证回退后报告不包含被回退内容
> 5. 重新综合：验证新 SynthesisGenerated 事件、节点数减少
>
> **30 秒跑完所有集成测试，比手工验证快 100 倍。**

## Phase 5: 生产就绪（Week 5）

### 5.1 会话数据持久化（2 天）

**现实意义**:

> **问题场景**: 用户花了 15 分钟做了一次深度分析（3 轮对话，每轮都用了 gpt-4o），生成了一份 2000 字的报告。总花费约 $0.15。第二天 Docker 容器自动重启（或运维执行 `docker compose down && docker compose up`）——**所有历史记录全部消失**。用户打开前端，看到空白对话列表。他不仅丢失了分析结果，丢失了花的钱，还丢失了对系统的信任。
>
> **改造后**: 会话数据从 `/tmp`（内存文件系统）迁移到 Docker 命名卷（持久化到宿主磁盘）。容器重启后数据完好。这是**任何面向用户产品的底线**。

**技术方案**: 
- `BASE_DIR` 改为 `os.environ.get("SYNAPSEGRID_SESSIONS_DIR", "/tmp/synapse_sessions")`
- `docker-compose.yml` 中挂载命名卷到 `/data/sessions`
- 启动时检测孤儿数据库文件（上次异常关闭残留）并清理

**验收**: `docker compose down && docker compose up` 后历史数据仍在。

---

### 5.2 Docker 部署完善（2 天）

**现实意义**:

> **问题场景**: 一个新用户想试用 SynapseGrid。他按照 README 执行 `docker compose up`——但后端启动后前端访问 `localhost:5173` 报 CORS 错误，因为前端直接请求 `localhost:8000` 而浏览器认为跨域。用户不是前端开发者，不知道什么是 CORS，他只知道"这个项目跑不起来"，然后放弃了。
>
> **改造后**: 
> - Nginx 反向代理：前端 80 端口，`/api/*` 代理到后端 8000 端口——同域访问，无 CORS 问题
> - 多阶段 Docker 构建：前端构建产物大小从 200MB 降到 20MB
> - 健康检查：`docker compose ps` 能看到服务状态，运维人员一眼知道是否正常

**技术方案**: 
- `frontend/nginx.conf` 增加 `proxy_pass http://backend:8000` 代理 `/api/` 和 `/health`
- `frontend/Dockerfile` 改为多阶段构建（node build → nginx serve）
- `docker-compose.yml` 增加 `healthcheck` 配置

**验收**: `docker compose up` 后浏览器访问 `localhost` 正常显示；`/api/v1/health` 返回健康状态。

---

### 5.3 环境配置标准化（1 天）

**现实意义**:

> **问题场景**: 当前 `session_manager.py` 中清理延迟硬编码为 300 秒，快照间隔硬编码为 50 个事件。如果生产环境需要调大清理延迟（保留历史记录更久供审计），或者调小快照间隔（更频繁的快照意味着更快的状态恢复），只能改代码重新部署。这对运维人员是不可接受的。
>
> **改造后**: 所有可调参数都通过环境变量配置，在 `.env.example` 中有完整文档。运维人员只需修改 `.env` 文件，无需改代码。

**技术方案**: 
- `SYNAPSEGRID_CLEANUP_DELAY`（默认 300）
- `SYNAPSEGRID_SNAPSHOT_INTERVAL`（默认 50）
- `SYNAPSEGRID_SESSIONS_DIR`（默认 `/tmp/synapse_sessions`）
- `.env.example` 完整文档化

**验收**: 修改 `.env` 后重启，清理延迟和快照间隔生效。

## 五、风险管控

| 风险 | 概率 | 影响 | 应对 |
