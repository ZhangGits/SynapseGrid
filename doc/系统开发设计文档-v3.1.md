## 一、系统概述

### 1.1 项目定位

SynapseGrid 是一个**可审计的事件溯源 AI Agent 系统**，核心特征：

- **事件溯源（Event Sourcing）**：所有操作以不可变事件日志形式持久化到 SQLite
- **三阶段 Agent 流水线**：Research → Verification → Synthesis
- **可回退分析**：支持对单个分析要点（Finding）进行回退，基于状态重建生成新报告
- **双向交互**：自然语言输入 ↔ 图形化画布输出，支持双向联动
- **密码学完整性**：Merkle 树 + HMAC 签名确保报告不可篡改

### 1.2 技术栈

| 层级 | 技术选型 |
|------|---------|
| 前端 | React 18 + TypeScript + Vite + Tailwind CSS |
| 后端 | Python 3.11 + FastAPI + Uvicorn |
| 数据 | SQLite（WAL 模式，每会话独立数据库）|
| 部署 | Docker + Docker Compose |
| AI | OpenAI GPT-4o-mini（可配置）|

### 2.3 关键模块详解

#### 2.3.1 事件存储（Event Store）

**设计原则**：不可变事件日志，每个会话独立 SQLite 数据库。

**核心表结构**：

```sql
CREATE TABLE events (
    version     INTEGER PRIMARY KEY,  -- 自增版本号
    event_type  TEXT NOT NULL,        -- 事件类型
    actor       TEXT NOT NULL,        -- 执行者（LlmConfig JSON 或字符串）
    payload     TEXT NOT NULL,        -- 事件载荷（JSON）
    metadata    TEXT NOT NULL,        -- 元数据（JSON）
    event_hash  TEXT NOT NULL,        -- 事件哈希（SHA-256）
    created_at  TEXT NOT NULL         -- ISO 8601 时间戳
);

CREATE TABLE snapshots (
    version     INTEGER PRIMARY KEY,  -- 快照版本
    state       TEXT NOT NULL,        -- 序列化状态
    merkle_root TEXT NOT NULL,        -- Merkle 根哈希
    created_at  TEXT NOT NULL
);
```

**SQLite 优化参数**（生产环境）：

| 参数 | 值 | 说明 |
|------|-----|------|
| journal_mode | WAL | 读写不阻塞 |
| wal_autocheckpoint | 2000 | 每 2000 页检查点 |
| cache_size | -64000 | 64MB 页面缓存 |
| synchronous | NORMAL | WAL 模式下足够安全 |
| busy_timeout | 5000 | 等待锁释放 5 秒 |

**事件类型枚举**：

| 事件类型 | 触发时机 | 关键载荷 |
|----------|---------|---------|
| TaskStarted | 用户提交任务 | prompt, template, output_mode |
| FindingReported | Research Agent 发现要点 | finding（claim, evidence, confidence）|
| FindingValidated | Verification Agent 验证通过 | finding_id |
| ConflictDetected | Verification Agent 发现矛盾 | conflict（type, severity, finding_a, finding_b）|
| SynthesisGenerated | Synthesis Agent 生成报告 | markdown_content, enriched_markdown, canvas_schema |
| FindingRolledBack | 用户回退要点 | finding_id, reason |
| SynthesisRestored | 回退后重新生成报告 | markdown_content, rolled_back_finding |

#### 2.3.2 会话管理器（Session Manager）

**核心设计**：单写者队列（Single-Writer Queue）

每个会话有一个独立的 `asyncio.Queue`，只有一个协程（`_writer_loop`）从队列中取出事件并写入 SQLite。这消除了多协程竞争写锁的问题。

```python
class SessionContext:
    session_id: str
    event_store: SessionEventStore
    projection: SessionStateProjection
    lineage: SessionLineageGraph
    write_queue: asyncio.Queue  # 单写者队列
    writer_task: asyncio.Task   # 写者协程
```

**异常恢复**：写者协程崩溃时自动重启（最多 3 次），超过次数后通知所有等待中的 future。

#### 2.3.3 状态投影（State Projection）

事件驱动的内存状态重建：

```python
class SessionStateProjection:
    def apply(self, event: Event) -> None:
        # 根据事件类型更新内存状态
        if event.event_type == "FindingReported":
            self.state["findings"][fid] = finding
        elif event.event_type == "FindingRolledBack":
            self.state["findings"][fid]["rolled_back"] = True
        # ... 其他事件类型
```

状态包含：
- `task`: 原始任务信息
- `findings`: 所有分析要点（含 rolled_back 标记）
- `conflicts`: 检测到的矛盾
- `agent_traces`: Agent 执行追踪
- `merkle_root`: 当前 Merkle 根哈希

#### 2.3.4 Merkle 树

每个事件的 `event_hash` 构成 Merkle 树的叶子节点，根哈希存储在投影状态中。

```python
def merkle_root(hashes: list[str]) -> str:
    """计算 Merkle 根哈希"""
    if not hashes:
        return "sha256:" + hashlib.sha256(b"").hexdigest()
    # 两两配对哈希，直到只剩一个
    while len(hashes) > 1:
        next_level = []
        for i in range(0, len(hashes), 2):
            left = hashes[i]
            right = hashes[i + 1] if i + 1 < len(hashes) else left
            combined = hashlib.sha256((left + right).encode()).hexdigest()
            next_level.append(combined)
        hashes = next_level
    return "sha256:" + hashes[0]
```

#### 2.3.5 HMAC 签名

使用 `SYNAPSEGRID_HMAC_KEY` 环境变量对报告内容进行签名：

```python
msg = f"{session_id}|{merkle_root}|{event_count}|{markdown}".encode("utf-8")
digest = hmac.new(HMAC_KEY, msg, hashlib.sha256).hexdigest()
signature = f"hmac:{digest}"
```

签名嵌入 YAML 审计头：
```yaml
---
# synapsegrid_audit
session_id: <session_id>
merkle_root: <merkle_root>
event_count: <event_count>
content_signature: <hmac_signature>
---
```

### 2.4 Agent 层详解

#### 2.4.1 Research Agent

**职责**：分析用户 prompt，提取结构化分析要点（Findings）。

**输出格式**（JSON）：
```json
{
  "findings": [
    {
      "id": "f1",
      "claim": "分析结论...",
      "evidence": "支持证据...",
      "confidence": 0.85,
      "source": "来源引用..."
    }
  ]
}
```

#### 2.4.2 Verification Agent

**职责**：
1. 验证每个 finding 的可信度
2. 检测 findings 之间的矛盾

**输入优化（TOON 格式）**：
```
@f1|0.85|该公司营收增长20%
@f2|0.72|净利润下降5%
```

比 JSON 格式节省约 40% Token（去除引号、缩进、字段名）。

**输出**：
- `validated`: 通过验证的 finding ID 列表
- `conflicts`: 矛盾列表（`{type, severity, finding_a, finding_b}`）

#### 2.4.3 Synthesis Agent

**职责**：基于验证后的 findings 生成综合报告。

**标记化输出**（Provenance 标签）：
- `<provenance f1>`: 引用 finding f1，渲染为可点击下划线
- `<data 15%>`: 高亮数据，渲染为蓝色加粗
- `<conflict f1 f2>`: 标记矛盾区域，渲染为警告卡片

**输出模式**：
- `markdown`: 纯 Markdown 文本
- `canvas`: 结构化画布（JSON Schema），支持前端渲染为交互式报告

### 2.5 回退算法

**旧方式（已废弃）**：文本过滤（正则替换），存在 `f1` 误匹配 `f10` 等问题。

**新方式（当前）**：基于状态重建

1. 写入 `FindingRolledBack` 事件
2. 从 Projection 获取未回退的 findings（`state_after`）
3. 调用 `SynthesisAgent.run()` 传入**未回退的 findings 列表**
4. Synthesis Agent 基于这些 findings 重新生成完整报告
5. 在报告中插入 Rollback Notice
6. 写入 `SynthesisRestored` 事件

**优势**：
- 无文本误匹配问题
- 报告逻辑一致性保证
- 支持任意次数的回退和重新综合

### 4.2 滑动窗口速率限制

```python
class SimpleRateLimiter:
    def __init__(self, max_requests: int = 10, window_seconds: int = 60):
        self.max_requests = max_requests
        self.window_seconds = window_seconds
        self._records: dict[str, list[float]] = {}

    def is_allowed(self, ip: str) -> bool:
        now = time.time()
        window_start = now - self.window_seconds
        records = [t for t in self._records.get(ip, []) if t > window_start]
        if len(records) >= self.max_requests:
            self._records[ip] = records
            return False
        records.append(now)
        self._records[ip] = records
        return True
```

### 4.4 成本估算算法

```python
def _estimate_verification_cost(model: str, finding_count: int) -> float:
    """Verification: 输入 200 tokens/finding + 输出 300 tokens"""
    avg_price = _get_avg_price(model) / 1000
    return round((finding_count * 200 + 300) * avg_price, 6)

def _estimate_synthesis_cost(model: str, finding_count: int) -> float:
    """Synthesis: 输入 300 tokens/finding + 输出 2000 tokens"""
    avg_price = _get_avg_price(model) / 1000
    return round((finding_count * 300 + 2000) * avg_price, 6)
```

## 七、安全设计

### 7.1 已实施的安全措施

| 措施 | 实现 |
|------|------|
| HMAC 签名 | `post_processor.py` 对报告进行 HMAC-SHA256 签名 |
| CORS 限制 | `main.py` 从环境变量读取允许来源 |
| 密钥长度校验 | 启动时强制检查 HMAC 密钥 >=32 字节 |
| 速率限制 | `SimpleRateLimiter` 每 IP 10 次/分钟 |
| 请求体限制 | Prompt 长度限制 8000 字符 |
| 不可变事件日志 | SQLite WAL 模式，事件不可篡改 |

### 7.2 已知限制

| 限制 | 说明 |
|------|------|
| 速率限制内存存储 | 重启后计数器清零，生产环境建议换 Redis |
| 默认 HMAC 密钥 | 开发环境使用默认密钥，生产环境必须修改 |
| SQLite 单节点 | 不支持多节点共享会话数据 |
